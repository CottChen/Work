## 1. MySQL 三日志系统

MySQL 采用三日志协同工作机制，分别承担不同的持久化职责。理解这三个日志的区别与联系，是掌握 MySQL 持久化机制的关键。

### 1.1 binlog (Binary Log)

**解决的问题**：
- 主从复制的数据同步
- 时间点恢复 (PITR, Point-In-Time Recovery)
- 数据归档与审计

**原理**：
binlog 是 MySQL Server 层的日志，与存储引擎无关。它记录所有对数据库结构 (DDL) 和数据 (DML) 的变更操作，采用追加写入方式，文件写满后切换新文件，形成日志序列。

**记录格式**：

| 格式 | 说明 | 优点 | 缺点 |
|------|------|------|------|
| **STATEMENT** | 记录原始 SQL 语句 | 日志量小 | 某些函数 (如 NOW()) 无法复现 |
| **ROW** | 记录每行数据的变更 | 数据一致性高 | 批量更新时日志量大 |
| **MIXED** | 默认格式，根据情况切换 | 平衡安全与效率 | 行为不易预测 |

**写入时机**：事务提交后写入（两阶段提交的第二阶段）

**详细例子**：

```sql
-- 假设执行以下事务
BEGIN;
UPDATE accounts SET balance = balance - 100 WHERE id = 1;
UPDATE accounts SET balance = balance + 100 WHERE id = 2;
COMMIT;
```

**ROW 格式下的 binlog 内容**：
```
#240101 12:00:00 server id 1  end_log_pos 1234
# at 4
#240101 12:00:00 server id 1  end_log_pos 5678
# Table_map: `test`.`accounts` mapped to number 100
# at 9
#240101 12:00:00 server id 1  end_log_pos 1234
# Update_rows: table id 100 flags: STMT_END_F
### UPDATE test.accounts
### WHERE
###   @1=1
###   @2=1000  -- 原余额
### SET
###   @1=1
###   @2=900   -- 新余额
### WHERE
###   @1=2
###   @2=2000  -- 原余额
### SET
###   @1=2
###   @2=2100  -- 新余额
```

**时序图**：
```
客户端     →    MySQL Server   →   存储引擎 (InnoDB)   →   binlog 文件
  |                |                    |                    |
  |--- BEGIN ----->|                    |                    |
  |                |                    |                    |
  |--- UPDATE ---->|--- 执行修改 ------>| (内存中修改)        |
  |                |                    |                    |
  |--- COMMIT ---->|                    |                    |
  |                | (两阶段提交)        |                    |
  |                |------ 阶段 1 ------>|                    |
  |                |<-------------------| (prepare 确认)      |
  |                |                    |                    |
  |                |------ 阶段 2 ------>|                    |
  |                |<-------------------| (commit 确认)       |
  |                |                    |                    |
  |                | (写入 binlog)      |                    |
  |                |--------------------------------------->|
  |                |                    |                    |
  |<-- 成功 -------|                    |                    |
```

**局限性**：
- 不包含未提交的事务
- STATEMENT 格式在某些场景下无法保证主从数据一致
- 大事务会导致 binlog 写入延迟

### 1.2 redo log (重做日志)

**解决的问题**：
- 崩溃恢复 (Crash Recovery)
- WAL (Write-Ahead Logging) 实现
- 确保 ACID 中的持久性 (Durability)

**原理**：
redo log 是 InnoDB 存储引擎特有的物理日志，记录"在某个数据页上做了什么修改"。采用循环写入方式，空间固定，写满后从头覆盖已刷盘的日志。

**关键特性**：
- **物理日志**：记录的是数据页的物理修改
- **循环写入**：固定大小，循环使用
- **顺序写入**：高性能，避免随机 I/O
- **WAL 技术**：先写日志，后写数据页

**两阶段提交详解**：

为什么需要两阶段提交？核心是为了保证 binlog 和 redo log 的一致性。

```
完整提交流程：

┌─────────────────────────────────────────────────────────┐
│  1. 执行器调用引擎接口执行事务                            │
│     - 在内存 Buffer Pool 中修改数据页                     │
│     - 生成 redo log (状态：prepare)                      │
└─────────────────────────────────────────────────────────┘
                            ↓
┌─────────────────────────────────────────────────────────┐
│  2. 准备阶段 (Prepare Phase)                            │
│     - 写入 redo log，标记为 prepare 状态                  │
│     - fsync 落盘确保持久化                               │
│     - 此时事务尚未提交，处于中间状态                      │
└─────────────────────────────────────────────────────────┘
                            ↓
┌─────────────────────────────────────────────────────────┐
│  3. 写入 binlog                                         │
│     - Server 层将整个事务记录到 binlog                   │
│     - fsync 落盘                                        │
│     - binlog 写入成功后，事务视为已提交                   │
└─────────────────────────────────────────────────────────┘
                            ↓
┌─────────────────────────────────────────────────────────┐
│  4. 提交阶段 (Commit Phase)                             │
│     - 写入 redo log，标记为 commit 状态                   │
│     - fsync 落盘                                        │
│     - 返回客户端提交成功                                 │
└─────────────────────────────────────────────────────────┘
```

**如果顺序反过来会怎样？**

```
错误顺序：先写 binlog，再写 redo log

场景：事务在 binlog 写入后、redo log 写入前崩溃

结果：
- binlog 中有该事务记录
- redo log 中无该事务记录
- 主从复制：从库会执行该事务（基于 binlog）
- 主库恢复：该事务丢失（redo log 无记录）
- 导致主从数据不一致！
```

**详细例子 - 完整数据变更流程**：

```sql
-- 事务 T1: 转账操作
BEGIN;
UPDATE accounts SET balance = 900 WHERE id = 1;  -- T1-A
UPDATE accounts SET balance = 1100 WHERE id = 2; -- T1-B
COMMIT;
```

**时间线分析**：

| 时间点 | 事件 | redo log 状态 | binlog 状态 | 内存数据页 |
|--------|------|---------------|-------------|------------|
| T0 | 事务开始 | - | - | 未修改 |
| T1 | 执行 T1-A | 写入 record-A (prepare) | 无 | Buffer Pool 修改 |
| T2 | 执行 T1-B | 写入 record-B (prepare) | 无 | Buffer Pool 修改 |
| T3 | 提交 - 阶段 1 | fsync (prepare 状态) | 无 | 未刷盘 |
| T4 | 提交 - 写 binlog | 保持 prepare | 写入事务记录 | 未刷盘 |
| T5 | 提交 - 阶段 2 | fsync (commit 状态) | fsync | 未刷盘 |
| T6 | 后台刷盘 | - | - | 刷入数据文件 |

**宕机恢复示例**：

假设在 T3 和 T4 之间（redo log 已 prepare，binlog 未写）发生宕机：

```
恢复流程：
1. 启动时读取 redo log
2. 发现 T1 处于 prepare 状态
3. 检查 binlog：
   - 如果 binlog 完整 → 重做 T1 (写入 commit 标记)
   - 如果 binlog 不完整 → 回滚 T1
```

**checkpoint 与刷盘**：

```
检查点机制：
- 定义：已安全刷盘到数据文件的 redo log 位置
- 作用：崩溃恢复时只需重做 checkpoint 之后的日志
- 类型：
  - Sharp Checkpoint：严格同步，性能差
  - Fuzzy Checkpoint：异步刷盘，MySQL 采用
```

**与其他机制的对比**：

| 特性 | redo log | binlog |
|------|----------|--------|
| 所属层级 | 存储引擎层 (InnoDB) | Server 层 |
| 日志类型 | 物理日志 (页修改) | 逻辑日志 (SQL/行变更) |
| 写入方式 | 循环写入 | 追加写入 |
| 空间管理 | 固定大小，循环覆盖 | 多文件，可归档 |
| 写入时机 | 执行时 | 提交时 |
| 主要用途 | 崩溃恢复 | 复制、PITR |

**局限性**：
- 仅 InnoDB 支持（其他引擎如 MyISAM 无此能力）
- 频繁 fsync 影响性能
- 大事务占用大量 redo log 空间

### 1.3 undo log (回滚日志)

**解决的问题**：
- 事务回滚 (ROLLBACK)
- MVCC (多版本并发控制)
- 崩溃后未完成事务的撤销

**原理**：
undo log 是逻辑日志，记录与数据操作相反的动作。每次数据变更都记录对应的反向操作，事务提交前不能删除。

**记录内容**：

| 操作 | undo log 记录 |
|------|---------------|
| INSERT | 删除主键值 |
| UPDATE | 原字段值 |
| DELETE | 完整行数据 |

**详细例子**：

```sql
-- 事务 T1
BEGIN;
INSERT INTO users (id, name) VALUES (1, 'Alice');  -- U1
UPDATE users SET name = 'Bob' WHERE id = 1;        -- U2
DELETE FROM users WHERE id = 1;                    -- U3

-- 对应的 undo log
U1: DELETE FROM users WHERE id = 1;
U2: UPDATE users SET name = 'Alice' WHERE id = 1;
U3: INSERT INTO users (id, name) VALUES (1, 'Bob');
```

**MVCC 中的应用**：

```
undo log 版本链：

行记录 header → 指向 undo log 版本 1
                    ↓
                undo log 版本 1 (name='Alice') → 指向 undo log 版本 2
                                                    ↓
                                                undo log 版本 2 (name='Bob')

事务 T2 读取时：
- 根据事务可见性规则
- 沿版本链找到可见的历史版本
- 实现非阻塞读 (快照读)
```

**生命周期**：

```
未提交事务：
- undo log 必须保留
- 可随时回滚

已提交事务：
- 不再需要回滚
- 但可能仍需 MVCC 读
- 无活跃事务引用后可清理

崩溃恢复：
- 扫描 redo log 找到未提交事务
- 沿 undo log 链执行回滚
```

**局限性**：
- 长事务导致 undo log 膨胀
- 大量历史版本影响读性能
- 需要 purge 线程定期清理

## 2. WAL (Write-Ahead Logging)

### 2.1 核心原理

**解决的问题**：
如何在保证数据持久性的前提下，最大化写入性能。

**原理**：

WAL 的核心思想可以用一句话概括：

> 在数据页写入磁盘之前，必须先将修改记录写入日志并落盘。

```
传统方式 vs WAL 方式：

传统方式 (Random I/O):
数据修改 → 随机位置写入数据文件 → fsync
问题：随机 I/O 性能差，每次修改都要磁盘寻道

WAL 方式 (Sequential I/O):
数据修改 → 顺序写入日志文件 → fsync → 异步刷数据页
优势：顺序 I/O 性能高，批量刷盘
```

**为什么 WAL 能保证持久性？**

```
ACID 持久性保证：

1. 事务提交前，redo log 已落盘
2. 即使立即宕机，数据页未刷盘也不影响
3. 恢复时重放 redo log 即可重建数据

数学归纳：
- 基础：空数据库，WAL 为空，一致
- 归纳：假设状态 S 一致，应用 WAL 记录到 S'
- 结论：任意崩溃后，重放 WAL 可达一致状态
```

---

### 2.2 PostgreSQL WAL 实现

**WAL 文件结构**：

```
pg_wal/
├── 000000010000000000000001  # WAL 段文件 (默认 16MB)
├── 000000010000000000000002
├── ...
└── 0000000100000000000000FF
```

**WAL 记录格式**：

```c
// 简化版 WAL 记录头
typedef struct XLogRecord {
    uint32 xl_tot_len;    // 记录总长度
    TransactionId xl_xid; // 事务 ID
    uint32 xl_prev;       // 上一条记录位置
    uint8  xl_info;       // 信息位
    RmgrId xl_rmid;       // 资源管理器 ID
    // 后面跟随实际数据
} XLogRecord;
```

**写入流程**：

```
1. 事务修改数据页（在共享缓冲区）
2. 生成 WAL 记录（包含：操作类型、数据页 ID、修改内容）
3. WAL 缓冲区积累
4. 提交时 WAL Writer 进程刷盘
5. 检查点定期将脏页刷入数据文件
```

**PostgreSQL vs MySQL 对比**：

| 特性 | PostgreSQL WAL | MySQL redo log + binlog |
|------|----------------|-------------------------|
| 日志类型 | 统一 WAL | 双日志系统 |
| 归档能力 | 原生支持 (archive_mode) | 依赖 binlog |
| 复制基础 | WAL 流复制 | binlog 复制 |
| 崩溃恢复 | WAL 重放 | redo log 重放 |
| 日志切换 | 自动段切换 | 循环覆盖 |
| Checkpoint | 可配置间隔 | 自适应 |

架构差异：

PostgreSQL:

## 3. MongoDB Journal 机制

### 3.1 作用与原理

**解决的问题**：
- WiredTiger 存储引擎的崩溃恢复
- 确保写操作在掉电后不丢失

**原理**：

MongoDB 的 journal 机制与 MySQL redo log 类似，但实现细节有所不同。

```
Journal 文件结构：

每个 journal 文件 100MB，预分配空间
┌─────────────────────────────────────┐
│ Journal Header (元数据)              │
├─────────────────────────────────────┤
│ Journal Entry 1                     │
│  - 事务头                           │
│  - 操作列表                         │
│  - 校验和                           │
├─────────────────────────────────────┤
│ Journal Entry 2                     │
└─────────────────────────────────────┘
```

**写入流程**：

```
1. 客户端写入请求
2. 修改内存中的 WiredTiger Cache
3. 写入 Journal（组提交优化）
4. Journal fsync 落盘（默认 100ms）
5. 定期 Checkpoint 将数据写入数据文件
```

**组提交 (Group Commit)**：

```
时间窗口内的多个写入合并：

T0: 事务 A 到达 → 等待
T1: 事务 B 到达 → 等待
T2: 事务 C 到达 → 等待
T3: Journal 间隔到 → 一次性写入 A+B+C → fsync

优势：减少 fsync 次数，提高吞吐量
代价：增加少量延迟
```

---

### 3.2 与 MySQL redo log 对比

| 特性 | MySQL redo log | MongoDB Journal |
|------|----------------|-----------------|
| 文件大小 | 固定 (innodb_log_file_size) | 100MB 自动扩展 |
| 写入触发 | 事务提交 | 定时 (默认 100ms) |
| 组提交 | 自动 | 可配置间隔 |
| 恢复粒度 | 事务级 | 操作级 |
| 与数据文件关系 | 独立机制 | WiredTiger 集成 |
| Checkpoint | 后台页刷盘 | 定期快照 |

**详细对比 - 数据变更流程**：

```
MySQL 事务提交：
┌──────┐   ┌──────────┐   ┌────────┐   ┌────────┐
│Client│   │Executor  │   │InnoDB  │   │RedoLog │
└──┬───┘   └────┬─────┘   └───┬────┘   └───┬────┘
   │COMMIT     │              │            │
   │──────────>│              │            │
   │           │ (1)prepare   │            │
   │           │─────────────>│            │
   │           │              │ write log  │
   │           │              │───────────>│
   │           │              │            │ fsync
   │           │              │<───────────│
   │           │ (2)binlog    │            │
   │           │─────────────>│            │
   │           │              │ fsync      │
   │           │<─────────────│            │
   │           │ (3)commit    │            │
   │           │─────────────>│            │
   │           │              │ write log  │
   │           │              │───────────>│
   │<----------│              │            │
   │OK         │              │            │
   │           │              │            │

MongoDB 写入：
┌──────┐   ┌──────────┐   ┌─────────────┐   ┌────────┐
│Client│   │mongod    │   │WiredTiger   │   │Journal │
└──┬───┘   └────┬─────┘   └──────┬──────┘   └───┬────┘
   │write      │                 │              │
   │──────────>│                 │              │
   │           │ 修改 cache      │              │
   │           │────────────────>│              │
   │           │                 │ 记录 journal │
   │           │                 │─────────────>│
   │           │                 │              │
   │           │ (100ms 后)      │              │
   │           │                 │              │ fsync
   │           │                 │─────────────>│
   │           │<────────────────│              │
   │<----------│                 │              │
   │ack        │                 │              │
```

## 4. Checkpoint 机制

### 4.1 什么是 Checkpoint

**解决的问题**：
- 限制崩溃恢复时间
- 控制日志文件大小
- 回收旧日志空间

**原理**：

Checkpoint 是数据库的一个"安全点"，表示该点之前的所有内存修改都已持久化到数据文件。

```
恢复时间优化：

无 Checkpoint:
崩溃后需要重放所有日志（可能数 GB）

有 Checkpoint:
只需重做最后一个 Checkpoint 之后的日志
恢复时间 = O(日志间隔) 而非 O(总日志量)
```

**Checkpoint 类型**：

| 类型 | 描述 | 性能影响 | 代表数据库 |
|------|------|----------|------------|
| Sharp | 阻塞所有操作直到刷盘完成 | 高 | 早期数据库 |
| Fuzzy | 后台异步刷盘，不阻塞 | 低 | MySQL, PostgreSQL |
| Non-blocking | 完全无锁 | 极低 | 现代数据库 |

---

### 4.2 MySQL Checkpoint 实现

**Fuzzy Checkpoint 机制**：

```
MySQL 不等待所有脏页刷盘，只确保部分脏页落盘。

关键参数：
- innodb_max_dirty_pages_pct: 触发刷盘的脏页比例（默认 75%）
- innodb_io_capacity: 每秒 I/O 操作数
- innodb_io_capacity_max: 刷盘时最大 I/O 操作数
```

**Checkpoint 位置追踪**：

```
┌─────────────────────────────────────────┐
│           Redo Log (循环)                │
│                                          │
│  checkpoint_pos → ●                      │
│                     ↓                    │
│  已刷盘区域  │   未刷盘区域               │
│  (可覆盖)   │   (保留)                   │
│                                          │
│  ←─────────────── 可用空间 ───────────→  │
└─────────────────────────────────────────┘
```

**后台刷盘线程**：

```python
# 简化版刷盘逻辑
def page_cleaner_thread():
    while True:
        # 1. LRU 扫描找脏页
        dirty_pages = scan_lru_list()

        # 2. 如果超过阈值，触发刷盘
        if len(dirty_pages) > max_dirty_pages:
            flush_pages(dirty_pages[:io_capacity])

        # 3. 更新 checkpoint 位置
        update_checkpoint_pos()

        sleep(1)  # 每秒检查
```

### 4.3 PostgreSQL Checkpoint 实现

**配置参数**：

```postgresql
# postgresql.conf
checkpoint_timeout = 15min      # 检查点间隔
checkpoint_completion_target = 0.5  # 刷盘时间占比
max_wal_size = 1GB              # WAL 最大大小
min_wal_size = 80MB             # WAL 最小大小
```

**Checkpoint 触发条件**：

```
1. 时间触发：checkpoint_timeout 到期
2. 空间触发：WAL 使用量超过 max_wal_size
3. 手动触发：CHECKPOINT; 命令
```

**Checkpoint 流程**：

```
1. Checkpointer 进程唤醒
2. 记录 checkpoint 开始位置 (REDO 起点)
3. 后台写入所有脏页
4. 写入 checkpoint 记录到 WAL
5. 更新控制文件中的 checkpoint 位置
6. 通知 WAL 可以截断旧日志
```

**性能优化**：

```
checkpoint_completion_target = 0.9 的含义：

假设 checkpoint_timeout = 15min
目标：用 15 × 0.9 = 13.5min 完成刷盘

效果：
- 值越大，I/O 越平滑，平时压力小
- 值越小，I/O 越集中，平时压力大
```

### 4.4 MongoDB Checkpoint 实现

**WiredTiger Checkpoint**：

```
MongoDB 的 WiredTiger 引擎使用"快照式"checkpoint。

特点：
- 定期（默认 60 秒）创建一致性快照
- 单线程执行，避免锁竞争
- 支持多个 checkpoint 共存
```

**Checkpoint 文件格式**：

```
WiredTiger.turtle 文件记录 checkpoint 元数据：
{
  "checkpoint_name": "WiredTigerCheckpoint.123",
  "checkpoint_time": "2024-01-01T12:00:00Z",
  "root_page_offset": 123456,
  "wal_start_offset": 789012
}
```

**恢复时的 checkpoint 选择**：

```
启动时：
1. 读取 WiredTiger.turtle
2. 找到最新的有效 checkpoint
3. 验证 checkpoint 一致性
4. 从该点开始重放 journal
```

## 5. 宕机恢复流程

### 5.1 正常 shutdown 流程

**MySQL 正常关闭**：

```
1. 停止接受新连接
2. 等待所有活跃事务完成
3. 刷新所有脏页到数据文件
4. 写入 shutdown 标记到 redo log
5. 关闭所有后台线程
6. 关闭文件句柄
```

**PostgreSQL 正常关闭**：

```
1. 停止接受新连接 (smart shutdown)
2. 等待所有后端进程退出
3. 执行最终 checkpoint
4. 写入 shutdown 记录到 WAL
5. 更新 pg_control 控制文件
6. 退出 postmaster 进程
```

**MongoDB 正常关闭**：

```
1. 刷新所有写入到磁盘
2. 创建最终 checkpoint
3. 关闭 journal
4. 释放文件锁
5. 退出进程
```

### 5.2 崩溃检测机制

**如何判断是崩溃而非正常关闭？**

```
MySQL:
- 检查 redo log 头部/尾部的 checksum
- 查找是否有未完成的 checkpoint
- 验证 ibdata1 文件头部的 shutdown 标记

PostgreSQL:
- 读取 pg_control 文件
- 检查数据库状态字段：
  - DB_STATE_SHUTDOWN_OK: 正常
  - DB_STATE_IN_PRODUCTION: 崩溃
  - DB_STATE_IN_RECOVERY: 恢复中

MongoDB:
- 检查 journal 文件完整性
- 验证 WiredTiger.turtle 的 timestamp
- 检测文件锁是否异常释放
```

### 5.3 恢复过程详解

**MySQL 崩溃恢复**：

```
阶段 1: 分析 (Analysis)
┌─────────────────────────────────────┐
│ 1. 读取 redo log，构建事务列表       │
│ 2. 识别已提交但未刷盘的事务          │
│ 3. 识别未完成的事务（需回滚）        │
└─────────────────────────────────────┘
        ↓
阶段 2: 重做 (Redo)
┌─────────────────────────────────────┐
│ 1. 从 checkpoint 位置开始扫描        │
│ 2. 对每个已提交事务：                │
│    - 应用 redo log 记录到数据页      │
│    - 更新页的 LSN (Log Sequence Num)│
│ 3. 确保所有提交事务的数据落盘        │
└─────────────────────────────────────┘
        ↓
阶段 3: 撤销 (Undo)
┌─────────────────────────────────────┐
│ 1. 找到所有未提交事务                │
│ 2. 沿 undo log 链反向执行            │
│ 3. 清理事务相关的临时资源            │
└─────────────────────────────────────┘
```

**完整宕机恢复示例**：

```
假设场景：
- T1: 事务 A 开始 (xid=1001)
- T2: 事务 A 修改页 P1 (balance=1000→900)
- T3: 事务 A 修改页 P2 (balance=2000→2100)
- T4: Checkpoint (P1 已刷盘，P2 未刷)
- T5: 事务 A 提交 (redo: prepare → binlog → commit)
- T6: 事务 B 开始 (xid=1002)
- T7: 事务 B 修改页 P3 (balance=500→400)
- T8: 崩溃！

恢复过程：

1. 读取 checkpoint 位置 (T4)
2. 从 T4 开始扫描 redo log

   T4 之后：
   - P1 已刷盘 → 跳过
   - P2 未刷盘 → 重做 (apply redo record)
   - P3 未刷盘 → 重做 (apply redo record)

3. 检查事务状态：
   - 事务 A (1001): commit → 已完成，无需操作
   - 事务 B (1002): active → 需要回滚

4. 回滚事务 B:
   - 读取 undo log for xid=1002
   - 执行反向操作：P3 balance=400→500
   - 清理 undo log

5. 恢复完成，数据库启动
```

**时序图**：

```
崩溃恢复时序：

┌─────────┐  ┌─────────┐  ┌──────────┐  ┌─────────┐
│ 启动    │  │ 分析    │  │ 重做    │  │ 撤销    │
│ 检测    │  │ 阶段    │  │ 阶段    │  │ 阶段    │
└────┬────┘  └────┬────┘  └────┬─────┘  └────┬────┘
     │           │             │              │
     │ 读控制   │             │              │
     │ 文件     │             │              │
     │──────────>│             │              │
     │           │             │              │
     │           │ 扫描 redo   │              │
     │           │ 构建事务表   │              │
     │           │────────────>│              │
     │           │             │              │
     │           │             │ 重放已提交   │
     │           │             │ 事务日志     │
     │           │             │─────────────>│
     │           │             │              │
     │           │             │              │ 回滚未提交
     │           │             │              │ 事务
     │           │             │              │─────┐
     │           │             │              │     │
     │<──────────│             │              │     │
     │ 就绪     │             │              │     │
     │           │             │              │<────┘
```

## 6. 硬件配合

### 6.1 电池保护 (BBU - Battery Backup Unit)

**解决的问题**：
磁盘写入缓存 (Write Cache) 在掉电时会丢失数据。

**原理**：

```
RAID 卡/磁盘控制器通常启用写缓存以提高性能：

无 BBU:
写操作 → 控制器缓存 → 报告完成 → 异步写磁盘
             ↓
          掉电 → 缓存数据丢失！

有 BBU:
写操作 → 控制器缓存 → 报告完成 → BBU 保护 → 写磁盘
             ↓                    ↓
          掉电 → BBU 供电继续写盘 → 数据保存
```

**MySQL 配置建议**：

```sql
-- 检查 BBU 状态
SHOW VARIABLES LIKE 'innodb_flush_method';

-- 有 BBU 时可安全使用
SET GLOBAL innodb_flush_method = 'O_DIRECT';  -- 绕过 OS 缓存
SET GLOBAL innodb_flush_log_at_trx_commit = 2; -- 日志秒级刷盘

-- 无 BBU 时必须严格
SET GLOBAL innodb_flush_log_at_trx_commit = 1; -- 每次提交都 fsync
```

---

### 6.2 NVDIMM (非易失性内存)

**解决的问题**：
传统 DRAM 掉电丢失，SSD/HDD 写入延迟高。

**原理**：

```
存储层次结构：

CPU Cache (纳秒级，易失)
    ↓
DRAM (纳秒级，易失)
    ↓
NVDIMM (微秒级，非易失) ← 新型持久化介质
    ↓
SSD (微秒级，非易失)
    ↓
HDD (毫秒级，非易失)
```

**NVDIMM 类型**：

| 类型 | 技术 | 持久化方式 | 代表产品 |
|------|------|------------|----------|
| NVDIMM-N | NAND + DRAM | 掉电时 NAND 备份 | Intel Optane |
| NVDIMM-F | 纯 NAND | 直接持久化 | Intel Optane PMem |
| NVDIMM-P | 混合模式 | 可配置 | 下一代产品 |

**数据库适配**：

```
MySQL 8.0+ 支持 NVDIMM：

innodb_dedicated_server = ON  # 自动检测 NVDIMM
innodb_undo_log_encrypt = ON  # NVDIMM 上加密 undo

PostgreSQL 13+:
可配置表空间到 NVDIMM 挂载点
CREATE TABLESPACE fast_ts LOCATION '/mnt/nvdimm';
```

### 6.3 磁盘缓存策略

**WCE (Write Cache Enable) vs Write-Through**：

```
WCE (写缓存启用):
┌─────────┐     ┌─────────┐     ┌─────────┐
│  应用   │────>│ 磁盘缓存 │────>│  盘片   │
└─────────┘     └─────────┘     └─────────┘
     │              │                │
     │              │ (立即返回)     │ (异步)
     │<─────────────│                │
     │ OK           │                │

优势：低延迟
风险：掉电丢数据

Write-Through (直写):
┌─────────┐     ┌─────────┐     ┌─────────┐
│  应用   │─────────────────>│  盘片   │
└─────────┘     │ 磁盘缓存 │     └─────────┘
     │              │                │
     │              │ (仅缓冲)       │ (同步)
     │              │<───────────────│
     │<─────────────────────────────│
     │ OK (确认落盘)

优势：数据安全
代价：高延迟
```

**Linux I/O 调度器配置**：

```bash
# 查看当前调度器
cat /sys/block/sda/queue/scheduler
# 输出：[mq-deadline] kyber none

# SSD/NVMe 推荐 none (无调度，由设备自己管理)
echo none > /sys/block/sda/queue/scheduler

# HDD 推荐 mq-deadline
echo mq-deadline > /sys/block/sda/queue/scheduler

# 关闭 NCQ (如果稳定性问题)
echo 1 > /sys/block/sda/device/queue_depth
```

### 6.4 不同硬件配置下的持久化保证

**配置矩阵**：

| 硬件配置 | 推荐设置 | 持久性保证 | 性能 |
|----------|----------|------------|------|
| **HDD + 无 BBU** | innodb_flush_log_at_trx_commit=1 | 最强 | 最低 |
| **HDD + BBU** | innodb_flush_log_at_trx_commit=2 | 强 | 中 |
| **SSD + 无 BBU** | innodb_flush_log_at_trx_commit=1 | 最强 | 中 |
| **SSD + BBU** | innodb_flush_log_at_trx_commit=2 | 强 | 高 |
| **NVMe + NVDIMM** | innodb_flush_log_at_trx_commit=2 | 强 | 最高 |
| **云盘 (EBS)** | innodb_flush_log_at_trx_commit=1 | 依赖云厂商 | 中 |

**fsync 策略对比**：

```
innodb_flush_log_at_trx_commit 参数：

0: 每秒刷盘
   - 性能最好
   - 崩溃丢失 1 秒数据
   - 仅适合可容忍数据丢失场景

1: 每次提交刷盘 (默认)
   - 最安全
   - 性能受 fsync 限制
   - 金融/核心系统推荐

2: 每次提交写 OS 缓存，每秒刷盘
   - 平衡方案
   - OS 崩溃丢失 1 秒数据
   - 有 BBU 时安全
```

**性能测试数据**：

```

→ 1200 tpmC

配置 B: innodb_flush_log_at_trx_commit=2, HDD + BBU
→ 3500 tpmC

配置 C: innodb_flush_log_at_trx_commit=1, NVMe
→ 8500 tpmC

配置 D: innodb_flush_log_at_trx_commit=2, NVMe + NVDIMM
→ 15000 tpmC
```

## 总结

### 日志机制对比总表

| 特性 | MySQL | PostgreSQL | MongoDB | Redis |
|------|-------|------------|---------|-------|
| WAL 类型 | redo log | WAL | Journal | AOF |
| 归档日志 | binlog | WAL Archive | - | RDB |
| 回滚日志 | undo log | - (MVCC 内建) | - | - |
| Checkpoint | 后台刷新 | 定期 | 定期 (60s) | BGSAVE |
| 恢复方式 | Redo + Undo | WAL 重放 | Journal 重放 | AOF 重放 |
| 复制基础 | binlog | WAL 流复制 | Oplog | - |

### 核心技术要点

1. **WAL 是基石**：所有现代数据库都采用 Write-Ahead Logging
2. **Checkpoint 定边界**：限制恢复时间，回收日志空间
3. **两阶段提交保一致**：MySQL 用以确保 redo log 与 binlog 一致
4. **硬件决定上限**：BBU、NVDIMM 等硬件直接影响持久化策略

### 最佳实践建议

1. **生产环境**：innodb_flush_log_at_trx_commit=1（最安全）
2. **有 BBU 时**：可适当放松到 2，换取性能
3. **定期检查**：监控 redo log/WAL 使用率，避免空间耗尽
4. **备份策略**：日志不能替代备份，定期全量 + 增量备份
