# 分布式事务专题

## 专题导读

分布式事务是数据库技术发展史上最复杂、最核心的挑战之一。随着数据量增长和业务全球化，单一数据库无法满足需求，数据必须分布在多个节点甚至多个数据中心。如何保证跨节点数据一致性，成为制约系统发展的关键瓶颈。

本专题按照技术演进顺序，详细解析：
1. 两阶段提交 (2PC) - 分布式事务的基石
2. 三阶段提交 (3PC) - 解决 2PC 的阻塞问题
3. TCC - 业务层面的最终一致性方案
4. Percolator 事务模型 - Google Spanner/TiDB 的核心协议
5. Spanner TrueTime - 利用物理时钟保证全球一致性
6. MongoDB 事务演进 - 从单文档到多文档的探索

## 1. 两阶段提交 (2PC)

### 1.1 解决的问题

**背景**：单机数据库时代，事务在单一节点内完成，通过 WAL 和锁机制保证 ACID。但当业务需要分库分表后，问题出现了：

```
场景：用户 A 从北京库转账 1000 元到上海库

┌─────────────┐              ┌─────────────┐
│  北京数据库  │              │  上海数据库  │
│  A: 5000 元   │              │  B: 3000 元   │
└──────┬──────┘              └──────┬──────┘
       │                            │
       │  1. A 扣 1000               │
       │  2. 写入 binlog            │
       │     └── 此时宕机！          │
       │                            │
       │  ❌ 钱扣了但 B 没收到        │
       │     数据不一致！            │
       ↓                            ↓
```

**核心问题**：跨多个独立数据库执行操作时，如何保证"全成功"或"全失败"？

**2PC 的诞生**：1978 年，Gray 在论文中提出两阶段提交协议，成为分布式事务的事实标准。

---

### 1.2 原理详解

两阶段提交引入一个**协调者 (Coordinator)** 角色，其他数据库节点称为**参与者 (Participant)**。

```
角色定义:
┌─────────────────┐
│   Coordinator   │  ← 事务协调者（中央控制节点）
│   (协调者)      │     - 发起事务
│                 │     - 收集投票
│                 │     - 决定提交/回滚
└────────┬────────┘
         │ 协调
         ↓
┌─────────────────────────────────────────┐
│  Participant  │  Participant  │  ...   │  ← 参与者（数据节点）
│  (节点 A)     │  (节点 B)     │        │     - 执行本地事务
│  ┌─────────┐  │  ┌─────────┐  │        │     - 投票 Yes/No
│  │ 本地 DB  │  │  │ 本地 DB  │  │        │     - 等待最终决定
│  └─────────┘  │  └─────────┘  │        │
└─────────────────────────────────────────┘
```

#### 阶段一：准备阶段 (Prepare Phase)

协调者向所有参与者发送 PREPARE 消息：

```
┌─────────────┐
│ Coordinator │
└──────┬──────┘
       │ PREPARE T1
       ├──────────────────────────────┐
       ↓                              ↓
┌──────────────┐              ┌──────────────┐
│ ParticipantA │              │ ParticipantB │
│              │              │              │
│ 1. 执行本地 SQL │              │ 1. 执行本地 SQL │
│    UPDATE A   │              │    UPDATE B  │
│              │              │              │
│ 2. 写入 undo/redo log       │              │
│    (持久化)   │              │    (持久化)  │
│              │              │              │
│ 3. 获取资源锁 │              │    获取资源锁 │
│    (不提交！) │              │    (不提交！) │
│              │              │              │
│ 4. 返回投票  │              │    返回投票  │
└──────┬───────┘              └──────┬───────┘
       │                              │
       │ YES/NO                       │ YES/NO
       └──────────────┬───────────────┘
                      ↓
             ┌────────────────┐
             │ 协调者收集投票  │
             └────────────────┘
```

**参与者决策逻辑**：
| 情况 | 动作 | 返回 |
|------|------|------|
| 本地执行成功 | 写入日志，持有锁 | YES |
| 执行失败/锁冲突 | 回滚，释放锁 | NO |
| 超时 | 回滚，释放锁 | NO |

#### 阶段二：提交阶段 (Commit Phase)

根据投票结果决定提交或回滚：

```
情况 A：所有参与者投票 YES
┌─────────────┐
│ Coordinator │  收到所有 YES
└──────┬──────┘
       │ COMMIT T1
       ├──────────────────────────────┐
       ↓                              ↓
┌──────────────┐              ┌──────────────┐
│ ParticipantA │              │ ParticipantB │
│              │              │              │
│ 1. 正式提交事务│              │  正式提交事务 │
│    COMMIT    │              │    COMMIT    │
│              │              │              │
│ 2. 释放资源锁 │              │   释放资源锁 │
│              │              │              │
│ 3. 返回 ACK  │              │   返回 ACK   │
└──────┬───────┘              └──────┬───────┘
       │                              │
       └──────────────┬───────────────┘
                      ↓
             ┌────────────────┐
             │ 收到所有 ACK    │
             │ 事务完成！      │
             └────────────────┘


情况 B：任一参与者投票 NO
┌─────────────┐
│ Coordinator │  收到至少一个 NO
└──────┬──────┘
       │ ROLLBACK T1
       ├──────────────────────────────┐
       ↓                              ↓
┌──────────────┐              ┌──────────────┐
│ ParticipantA │              │ ParticipantB │
│              │              │              │
│ 1. 回滚事务   │              │   回滚事务   │
│    ROLLBACK  │              │   ROLLBACK   │
│              │              │              │
│ 2. 释放资源锁 │              │   释放资源锁 │
│              │              │              │
│ 3. 返回 ACK  │              │   返回 ACK   │
└──────┬───────┘              └──────┬───────┘
       │                              │
       └──────────────┬───────────────┘
                      ↓
             ┌────────────────┐
             │ 收到所有 ACK    │
             │ 事务已回滚！    │
             └────────────────┘
```

### 1.5 2PC 的局限性总结

| 问题 | 描述 | 影响 |
|------|------|------|
| **单点故障** | Coordinator 宕机导致事务无法完成 | 系统可用性降低 |
| **阻塞问题** | 参与者在等待期间持有锁，阻塞其他请求 | 性能下降，可能死锁 |
| **数据不一致** | 网络分区时可能出现部分提交 | 最坏情况下数据不一致 |
| **同步阻塞** | 所有参与者必须等待最慢节点 | 延迟取决于最慢节点 |
| **日志开销** | 每个参与者都要写 undo/redo log | 磁盘 I/O 压力大 |

---

### 1.6 与下一个技术的关系

2PC 的**阻塞问题**和**单点故障**催生了三阶段提交 (3PC)。

3PC 的核心改进：
- 增加一个**预提交 (Pre-Commit)** 阶段
- 引入**超时机制**，参与者超时后可以自主决定
- 降低阻塞概率（但不能完全消除）

---

## 2. 三阶段提交 (3PC)

### 2.1 解决的问题

3PC 针对 2PC 的两个核心问题进行改进：

**问题 1：协调者宕机导致永久阻塞**

```
2PC 场景:
┌─────────────┐    ┌───────────┐
│ Coordinator │    │Participant│
│             │    │           │
     │                │
     │ PREPARE        │
     │───────────────>│
     │                │
     │                │ 执行成功，返回 YES
     │                │ 持有锁，等待 COMMIT
     │                │
     │ ⚡ 宕机！        │
     │                │ ⏱️ 等待...
     │                │ ⏱️ 继续等待...
     │                │ ❓ 永远等不到响应

参与者陷入永久阻塞
```

**问题 2：参与者在不确定状态下无法自主决策**

```
2PC 的不确定窗口:
                  ┌──────────────────────┐
参与者在          │ 已返回 YES           │
这个窗口内        │ 未收到最终决定       │
无法自主决定      │ 不能提交也不能回滚   │
                  └──────────────────────┘
```

---

### 2.2 原理详解

3PC 将原来的两阶段扩展为三个阶段：

```
2PC:                        3PC:
┌─────────────────┐        ┌─────────────────┐
│ 1. Prepare      │        │ 1. CanCommit    │
│ 2. DoCommit     │        │ 2. PreCommit    │
└─────────────────┘        │ 3. DoCommit     │
                           └─────────────────┘

关键改进:
1. CanCommit 阶段：询问"你能提交吗？"（不执行）
2. PreCommit 阶段：预执行，类似 2PC 的 Prepare
3. DoCommit 阶段：正式提交
```

#### 阶段一：CanCommit（询问阶段）

协调者询问所有参与者是否能提交，参与者只回答，不执行任何操作：

```
┌─────────────┐
│ Coordinator │
└──────┬──────┘
       │ CanCommit T1?
       ├──────────────────────────────┐
       ↓                              ↓
┌──────────────┐              ┌──────────────┐
│ ParticipantA │              │ ParticipantB │
│              │              │              │
│ 1. 检查资源   │              │  检查资源     │
│    (不执行！) │              │  (不执行！)  │
│    - 有余额吗？│              │  - 账户正常？ │
│    - 能获取锁？│              │  - 能获取锁？ │
│              │              │              │
│ 2. 返回响应  │              │  返回响应    │
└──────┬───────┘              └──────┬───────┘
       │                              │
       │ YES/NO                       │ YES/NO
       └──────────────┬───────────────┘
                      ↓
             ┌────────────────┐
             │ 收集所有响应    │
             │ 全部 YES → 下一阶段
             │ 任一 NO → 中断  │
             └────────────────┘

注意：此阶段不执行 SQL，不写日志，不持有任何锁
```

#### 阶段二：PreCommit（预提交阶段）

类似 2PC 的 Prepare 阶段，但增加了超时机制：

```
┌─────────────┐
│ Coordinator │
└──────┬──────┘
       │ PreCommit T1
       ├──────────────────────────────┐
       ↓                              ↓
┌──────────────┐              ┌──────────────┐
│ ParticipantA │              │ ParticipantB │
│              │              │              │
│ 1. 执行本地 SQL│              │  执行本地 SQL │
│    UPDATE ... │              │  UPDATE ...  │
│              │              │              │
│ 2. 写 undo/redo log         │  写日志       │
│    [BEGIN]   │              │  [BEGIN]     │
│    [PREPARE] │              │  [PREPARE]   │
│              │              │              │
│ 3. 获取资源锁 │              │  获取资源锁  │
│              │              │              │
│ 4. 返回 ACK  │              │  返回 ACK    │
└──────┬───────┘              └──────┬───────┘
       │                              │
       └──────────────┬───────────────┘
                      ↓
             ┌────────────────┐
             │ 收到所有 ACK    │
             │ → PreCommit 成功
             │ → 进入 DoCommit │
             └────────────────┘

超时机制:
- 参与者设置超时计时器（如 30 秒）
- 超时未收到 PreCommit → 自动回滚
- 超时未收到 DoCommit → 自动提交（关键改进！）
```

#### 阶段三：DoCommit（正式提交阶段）

根据 PreCommit 结果执行最终操作：

```
情况 A：PreCommit 成功
┌─────────────┐
│ Coordinator │  所有 ACK
└──────┬──────┘
       │ DoCommit T1
       ├──────────────────────────────┐
       ↓                              ↓
┌──────────────┐              ┌──────────────┐
│ ParticipantA │              │ ParticipantB │
│              │              │              │
│ 1. 正式提交   │              │   正式提交   │
│    COMMIT    │              │   COMMIT     │
│              │              │              │
│ 2. 释放锁    │              │   释放锁     │
│              │              │              │
│ 3. ACK       │              │   ACK        │
└──────┬───────┘              └──────┬───────┘
       │                              │
       └──────────────┬───────────────┘
                      ↓
             事务完成！

情况 B：PreCommit 失败（任一 NO 或超时）
┌─────────────┐
│ Coordinator │  有 NO 或超时
└──────┬──────┘
       │ Abort T1
       ├──────────────────────────────┐
       ↓                              ↓
┌──────────────┐              ┌──────────────┐
│ ParticipantA │              │ ParticipantB │
│              │              │              │
│ 1. 回滚事务   │              │   回滚事务   │
│    ROLLBACK  │              │   ROLLBACK   │
│              │              │              │
│ 2. 释放锁    │              │   释放锁     │
│              │              │              │
│ 3. ACK       │              │   ACK        │
└──────┬───────┘              └──────┬───────┘
       │                              │
       └──────────────┬───────────────┘
                      ↓
             事务已回滚！
```

### 2.3 3PC 的关键改进：超时自主决策

3PC 最核心的改进是**超时后可以自主决策**：

```
┌─────────────────────────────────────────────────────────────┐
│                    3PC 超时决策矩阵                          │
├───────────────────┬─────────────────┬───────────────────────┤
│ 超时场景           │ 参与者状态       │ 自主决策              │
├───────────────────┼─────────────────┼───────────────────────┤
│ CanCommit 超时    │ 未收到 PreCommit │ 自动回滚 (安全)        │
├───────────────────┼─────────────────┼───────────────────────┤
│ PreCommit 超时    │ 已执行，未决定   │ 自动回滚 (安全)        │
├───────────────────┼─────────────────┼───────────────────────┤
│ DoCommit 超时     │ 已 PreCommit    │ ⚠️ 自动提交 (有风险！) │
└───────────────────┴─────────────────┴───────────────────────┘

为什么 DoCommit 超时可以自动提交？
答：因为 PreCommit 阶段所有参与者已达成一致，
    理论上协调者一定会发 DoCommit，
    超时只是网络问题，最终决定应该是提交。
```

### 2.5 3PC 仍然存在的问题

虽然 3PC 降低了阻塞概率，但仍存在问题：

#### 问题 1：网络分区导致数据不一致

```
场景：网络分区时，3PC 可能导致数据不一致

┌─────────────┐
│ Coordinator │
└──────┬──────┘
       │
  ═════╪══════════════════════════════════ 网络分区边界
       │                  │
       │         ┌────────┴────────┐
       │         │   ParticipantA  │
       │         │   ParticipantB  │
       │         │   (已 PreCommit)│
       │         └─────────────────┘
       │
  ⚡ 协调者与部分参与者失联

分区 A (协调者 + 部分参与者):
  → 超时后决定回滚

分区 B (已 PreCommit 的参与者):
  → 超时后自动提交

结果：部分提交，部分回滚 → 数据不一致！
```

#### 问题 2：自动提交的风险

```
场景：协调者实际决定回滚，但参与者超时自动提交

┌─────────────┐    ┌───────────┐
│ Coordinator │    │Participant│
│             │    │           │
     │                │
     │ PreCommit      │
     │───────────────>│
     │                │ 执行...ACK
     │<───────────────│
     │                │
     │ 发现某个参与者 NO
     │ 决定：回滚
     │                │
     │ ⚡ 网络延迟      │ Abort 消息未到
     │                │
     │                │ ⏱️ 超时！
     │                │ 自主决策：提交！
     │                │ COMMIT ← 与协调者决定相反！
     │                │
     ╳ 数据不一致！

这是 3PC 为降低阻塞付出的代价
```

### 2.6 与 2PC 的对比总结

| 特性 | 2PC | 3PC |
|------|-----|-----|
| 阶段数 | 2 | 3 |
| 阻塞问题 | 严重 | 降低但未消除 |
| 单点故障 | 影响大 | 影响降低 |
| 数据一致性 | 较好 | 网络分区时可能不一致 |
| 实现复杂度 | 低 | 中 |
| 实际应用 | XA 协议广泛使用 | 较少单独使用 |

### 2.7 与下一个技术的关系

3PC 的**数据不一致风险**促使人们寻找更可靠的方案。

TCC（Try-Confirm-Cancel）从业务层面重新思考问题：
- 不追求强一致性，接受最终一致性
- 将事务控制权交给业务代码
- 通过补偿机制保证最终状态正确

## 3. TCC (Try-Confirm-Cancel)

### 3.1 解决的问题

TCC 针对 2PC/3PC 的问题提出新思考：

**问题 1：数据库层面的强一致性代价太高**

```
2PC/3PC 的困境:
- 需要持有锁等待所有参与者
- 阻塞其他业务请求
- 跨数据中心延迟高
- 硬件故障影响大

业务思考：真的需要强一致性吗？

电商场景：
- 下单 → 扣库存 → 扣款
- 可以接受秒级最终一致
- 不需要数据库层面的强一致
```

**问题 2：数据库不知道业务语义**

```
数据库视角:
UPDATE inventory SET count = count - 1 WHERE id = 123;
→ 只是一行数据变化，不知道"库存"的业务含义

业务视角:
- 需要预留库存（不是真扣）
- 支付成功后再真扣
- 支付失败要释放

TCC 的思路：让业务代码控制事务！
```

---

### 3.2 原理详解

TCC 将事务拆分为三个业务方法：

```
┌─────────────────────────────────────────────────────────────┐
│                    TCC 三阶段业务含义                        │
├───────────────┬─────────────────────────────────────────────┤
│ 阶段          │ 业务动作                                    │
├───────────────┼─────────────────────────────────────────────┤
│ Try           │ 1. 检查业务条件（余额、库存等）              │
│ (尝试)        │ 2. 预留业务资源（冻结而非扣除）              │
│               │ 3. 返回成功/失败                            │
├───────────────┼─────────────────────────────────────────────┤
│ Confirm       │ 1. 确认执行                                │
│ (确认)        │ 2. 真正使用预留资源（扣款、扣库存）           │
│               │ 3. 释放预留（如果不用）                     │
│               │ ⚠️ 必须幂等！                               │
├───────────────┼─────────────────────────────────────────────┤
│ Cancel        │ 1. 取消执行                                │
│ (取消)        │ 2. 释放预留资源（解冻）                     │
│               │ 3. 回滚业务状态                            │
│               │ ⚠️ 必须幂等！                               │
└───────────────┴─────────────────────────────────────────────┘

TCC 流程:
     ┌─────────────┐
     │  开始事务    │
     └──────┬──────┘
            │
            ↓
     ┌─────────────┐
     │ Try 阶段     │  ← 所有参与者执行 Try
     │ (预留资源)   │     全部成功 → Confirm
     └──────┬──────┘     任一失败 → Cancel
            │
     ┌──────┴──────┐
     ↓             ↓
┌─────────┐   ┌─────────┐
│Confirm  │   │ Cancel  │
│ (确认)  │   │ (取消)  │
└─────────┘   └─────────┘
```

### 3.5 空回滚、幂等、悬挂问题处理

TCC 实现必须处理三个关键问题：

#### 问题 1：空回滚（Try 未执行，Cancel 先到达）

```
场景：Try 请求因网络丢失，Cancel 先到

┌──────────┐   ┌──────────┐
│Coordinator│   │Participant│
│          │   │          │
    │ Try       │
    │──────────>│
    │           │ ❌ 网络丢失，未到
    │           │
    │ 决定：取消  │
    │ Cancel    │
    │──────────>│
    │           │ ❓ Try 没执行，Cancel 来了
    │           │    怎么办？
    │           │
    │           │ 正确处理：
    │           │ 1. 记录 Cancel 已调用
    │           │ 2. 后续 Try 到达时拒绝
    │           │ 3. 返回成功（幂等）

代码示例:
public void cancel(String txId) {
    if (tryRecord.exists(txId)) {
        // 正常 Cancel，释放资源
        release(txId);
    } else {
        // 空回滚，记录已 Cancel
        cancelRecord.add(txId);
    }
}

public void try(String txId) {
    if (cancelRecord.exists(txId)) {
        // Try 晚到了，拒绝
        throw new TryAfterCancelException();
    }
    // 正常 Try 逻辑
}
```

#### 问题 2：幂等性（同一方法多次调用）

```
场景：Confirm 重复调用

┌──────────┐   ┌──────────┐
│Coordinator│   │Participant│
│          │   │          │
    │ Confirm   │
    │──────────>│
    │           │ ✓ 执行成功
    │<──────────│
    │           │
    │ 超时重试  │
    │ Confirm   │ ← 重复调用
    │──────────>│
    │           │ ❓ 已执行过，怎么办？
    │           │
    │           │ 正确处理：
    │           │ 1. 检查执行记录
    │           │ 2. 已执行则直接返回成功
    │           │ 3. 不重复执行

代码示例:
public void confirm(String txId) {
    if (confirmRecord.exists(txId)) {
        // 已执行，幂等返回
        return;
    }
    // 执行 Confirm 逻辑
    doConfirm(txId);
    // 记录已执行
    confirmRecord.add(txId);
}
```

#### 问题 3：悬挂（Cancel 先到，Try 后到）

```
场景：网络延迟导致 Try 晚于 Cancel

┌──────────┐   ┌──────────┐
│Coordinator│   │Participant│
│          │   │          │
    │ Try       │
    │           │ ⏱️ 网络拥堵
    │           │
    │ 决定：取消  │
    │ Cancel    │
    │──────────>│
    │           │ 执行 Cancel
    │           │ 记录已 Cancel
    │           │
    │           │ ⏱️ Try 终于到了...
    │ Try       │
    │──────────>│
    │           │ ❓ Cancel 已执行，Try 才到
    │           │
    │           │ 正确处理：
    │           │ 1. 检查 Cancel 记录
    │           │ 2. 拒绝执行 Try
    │           │ 3. 防止资源被错误预留
```

### 3.6 TCC 优缺点分析

#### 优点

| 优点 | 说明 |
|------|------|
| **无锁等待** | Try 阶段预留，不持有长锁 |
| **高性能** |  Confirm/Cancel 异步执行 |
| **业务可控** | 业务代码明确知道预留/释放逻辑 |
| **最终一致** | 允许短暂不一致，符合业务实际 |

#### 缺点

| 缺点 | 说明 | 影响 |
|------|------|------|
| **侵入性强** | 每个服务需实现 Try/Confirm/Cancel | 开发成本高 |
| **实现复杂** | 需处理幂等、空回滚、悬挂 | 容易出错 |
| **资源预留** | Try 阶段需预留资源 | 资源利用率低 |
| **不支持嵌套** | TCC 事务不能嵌套调用 | 场景受限 |

### 3.7 与上一个技术的关系

3PC 试图在**协议层面**解决阻塞问题，但仍存在网络分区时的数据不一致。

TCC 从**业务层面**重新思考：
- 不追求强一致性，接受最终一致性
- 将事务控制权交给业务
- 通过补偿机制保证最终状态

### 3.8 与下一个技术的关系

TCC 虽然灵活，但**侵入性太强**，每个业务都要实现三个方法。

Google Percolator 提出了新思路：
- 在**存储层面**实现分布式事务
- 业务代码无感知（像单机事务一样使用）
- 基于乐观锁 + 时间戳

## 4. Percolator 事务模型

### 4.1 解决的问题

Percolator 是 Google 在 2010 年发表的论文中提出的事务模型，用于解决：

**问题 1：2PC 性能太差**

```
2PC 的性能瓶颈:
- 每个参与者写日志 → 磁盘 I/O 大
- 持有锁等待 → 阻塞其他事务
- 同步提交 → 延迟高

Percolator 的思考:
能否像单机数据库一样高效？
→ 使用乐观锁，冲突时再回滚
```

**问题 2：业务不想处理复杂的 Try/Confirm/Cancel**

```
TCC 的困境:
- 每个接口要写三个方法
- 要处理幂等、空回滚、悬挂
- 业务代码被事务逻辑污染

Percolator 的目标:
- 业务代码无感知
- 像写单机代码一样简单
- 事务逻辑由存储层处理
```

---

### 4.2 原理详解

#### Percolator 架构

```
Percolator 构建在 Bigtable 之上:

┌─────────────────────────────────────────┐
│          Percolator 事务层               │
│  - 时间戳分配 (Oracle)                  │
│  - 乐观并发控制                         │
│  - 两阶段提交变体                       │
└─────────────────────────────────────────┘
                    ↓
┌─────────────────────────────────────────┐
│          Bigtable 存储层                 │
│  ┌─────────┬─────────┬─────────┐        │
│  │ Column  │ Column  │ Column  │        │
│  │  Data   │  Lock   │ Write   │        │
│  └─────────┴─────────┴─────────┘        │
└─────────────────────────────────────────┘

三列设计:
- Data 列：实际数据
- Lock 列：事务锁
- Write 列：提交记录
```

#### 时间戳分配机制

```
Percolator Oracle 服务:

┌─────────────┐
│   Oracle    │  ← 全局时间戳分配器
│   Service   │     - 单调递增
│             │     - 事务唯一标识
└──────┬──────┘
       │ get_timestamp()
       ↓
┌─────────────┐
│  T1 = 1001  │  ← 事务 1 的开始时间
│  T2 = 1002  │  ← 事务 2 的开始时间
│  T3 = 1003  │  ← 事务 3 的开始时间
└─────────────┘

关键：时间戳 = 事务版本 = 快照点
```

#### Snapshot Read（快照读）

```
读取规则：读取最后一个已提交的 Write 记录

示例:
时间线：
T=1001: 写入 (v1)
T=1002: 读取 → 看到 v1 (因为 1001 已提交)
T=1003: 写入 (v2)
T=1004: 读取 → 看到 v2 (因为 1003 已提交)

Bigtable 存储:
Row: user_123
┌─────────────────────────────────────────┐
│ Timestamp │ Column    │ Value           │
├─────────────────────────────────────────┤
│ 1003      │ Data      │ v2              │ ← 最新
│ 1003      │ Write     │ COMMITTED       │
│ 1001      │ Data      │ v1              │
│ 1001      │ Write     │ COMMITTED       │
└─────────────────────────────────────────┘

读取时：
1. 获取当前时间戳 T_read
2. 查找 < T_read 的最大已提交版本
3. 返回对应 Data 值
```

### 4.4 冲突检测与回滚

#### 场景：两个事务同时修改同一行

```
事务 T1 (start_ts=1001)     事务 T2 (start_ts=1002)
        │                          │
        │ read(A) ← 快照 T=1001    │
        │ 读到 A=1000              │
        │                          │ read(A) ← 快照 T=1002
        │                          │ 读到 A=1000 (T1 未提交)
        │                          │
        │ prewrite(A)              │
        │ Lock[A] = T1             │
        │ ✓ 成功                   │
        │                          │ prewrite(A)
        │                          │ ❌ 检测到 Lock!
        │                          │ 冲突！
        │                          │
        │                          │ 检查 Lock 时间戳:
        │                          │ Lock.ts (1001) > T2.start (1002)?
        │                          │ NO → Lock 是旧事务的
        │                          │
        │                          │ 尝试清理旧锁:
        │                          │ 检查 T1 是否已完成
        │                          │ → 未完成，T2 回滚
        │                          │
        │                          │ 回滚！
        │                          │ 重试（获取新时间戳）
        │                          │
        │ commit(A)                │
        │ ✓ 成功提交               │
```

### 4.5 局限性

| 问题 | 描述 | 影响 |
|------|------|------|
| **时钟依赖** | 需要外部时间戳服务 | 单点故障风险 |
| **写偏斜** | 某些场景需要额外处理 | 需要应用层注意 |
| **长事务问题** | 长事务持有锁时间长 | 增加冲突概率 |
| **二次写入** | Lock + Write 两次写入 | 写放大 |

### 4.6 与下一个技术的关系

Percolator 需要外部 Oracle 服务分配时间戳，但：
- **跨数据中心时，时间戳如何保证全局顺序？**
- **网络延迟导致的时间戳不同步怎么办？**

Google Spanner 的 TrueTime API 用 GPS+ 原子钟解决了这个问题！

## 5. Spanner TrueTime

### 5.1 解决的问题

在跨数据中心的分布式数据库中，时间同步是核心挑战：

**问题 1：时钟漂移**

```
不同数据中心的时钟存在漂移:

数据中心 A (北京):  10:00:00.000
数据中心 B (上海):  10:00:00.050  ← 快 50ms
数据中心 C (纽约):  09:59:59.980  ← 慢 20ms

如果直接用本地时间作为时间戳:
- T1 在北京提交: timestamp = 1000
- T2 在上海提交: timestamp = 1050
- 但 T2 实际发生在 T1 之前！
- 导致事务顺序错乱
```

**问题 2：NTP 精度不够**

```
NTP (网络时间协议) 的局限:
- 局域网精度：~10ms
- 广域网精度：~100ms
- 不可预测，可能跳变

Spanner 需求：
- 需要 μs 级别的精度
- 需要可预测的不确定度
- 不能跳变（单调性）
```

---

### 5.2 TrueTime 原理

TrueTime 的核心思想：**承认时间的不确定度**

```
TrueTime API 返回值:

TT.now() → [earliest, latest]

示例:
TT.now() = [1000ms, 1050ms]

含义:
- 当前真实时间在 1000ms 到 1050ms 之间
- 不确定度窗口 = 50ms
- earliest = 最乐观估计
- latest = 最保守估计
```

#### TrueTime 架构

```
┌─────────────────────────────────────────────────────────────┐
│                    TrueTime 架构                            │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│  ┌─────────────────────────────────────────────────┐       │
│  │              时间源层                            │       │
│  │  ┌───────────┐  ┌───────────┐  ┌───────────┐   │       │
│  │  │   GPS     │  │  原子钟   │  │  原子钟   │   │       │
│  │  │ (多卫星)   │  │ (本地备份) │  │ (冗余)   │   │       │
│  │  └─────┬─────┘  └─────┬─────┘  └─────┬─────┘   │       │
│  │        │              │              │           │       │
│  │        └──────────────┼──────────────┘           │       │
│  │                       ↓                         │       │
│  │  ┌─────────────────────────────────────────┐   │       │
│  │  │           Time Daemon (每台服务器)       │   │       │
│  │  │   - 持续同步 GPS+ 原子钟                  │   │       │
│  │  │   - 计算本地时钟误差 ε                  │   │       │
│  │  │   - 维护 [earliest, latest] 区间         │   │       │
│  │  └─────────────────────────────────────────┘   │       │
│  └─────────────────────────────────────────────────┘       │
│                           ↓                                 │
│  ┌─────────────────────────────────────────────────┐       │
│  │           TrueTime API                          │       │
│  │  TT.now() → [T - ε, T + ε]                      │       │
│  │  TT.after(t): 等待直到肯定 > t                   │       │
│  └─────────────────────────────────────────────────┘       │
│                                                             │
└─────────────────────────────────────────────────────────────┘

关键指标:
- ε (epsilon): 时钟不确定度半径
- Spanner 目标：ε < 7ms (典型值 2-3ms)
```

---

### 5.3 时间不确定度窗口的作用

#### 如何计算不确定度

```
ε = ε_osc + ε_sync + ε_drift

其中:
- ε_osc: 本地晶振误差（出厂校准）
- ε_sync: NTP 同步误差（测量得到）
- ε_drift: 自上次同步后的漂移（时间 × 漂移率）

示例:
假设:
- ε_osc = 1ms
- ε_sync = 2ms
- 漂移率 = 0.001ms/s
- 距离上次同步 = 100s

ε = 1 + 2 + (0.001 × 100) = 3.1ms

TT.now() = [T - 3.1ms, T + 3.1ms]
```

#### 不确定度窗口保证事务顺序

```
场景：事务 T1 提交后，T2 如何保证时间戳更大？

T1 提交:
  commit_ts1 = TT.now().latest = 1000 + 7 = 1007ms

T2 开始:
  TT.after(1007ms)  ← 等待直到 earliest > 1007
  等待...
  TT.now().earliest = 1008ms > 1007ms ✓
  start_ts2 = TT.now().earliest = 1008ms

结果:
  commit_ts1 = 1007ms
  start_ts2 = 1008ms
  1008 > 1007 ✓ 保证顺序！

关键：TT.after() 等待消除了不确定度窗口
```

---

### 5.4 事务提交如何依赖 TrueTime

#### 读写事务提交流程

```
时序图：TrueTime 保证事务顺序

┌─────────┐   ┌─────────┐   ┌───────────┐   ┌───────────┐
│ Client  │   │TrueTime │   │ 数据中心 A │   │ 数据中心 B │
└────┬────┘   └────┬────┘   └─────┬─────┘   └─────┬─────┘
     │            │               │               │
     │ 开始事务 T1 │               │               │
     │───────────>│               │               │
     │            │               │               │
     │ start_ts1  │               │               │
     │ = TT.now() │               │               │
     │ = [100,107]│               │               │
     │ 选 100      │               │               │
     │<───────────│               │               │
     │            │               │               │
     │ 读写操作... │               │               │
     │ (本地执行)  │               │               │
     │            │               │               │
     │ 准备提交    │               │               │
     │            │               │               │
     │ commit_ts1 │               │               │
     │ = TT.now().latest          │               │
     │ = [150,157]│               │               │
     │ 选 157     │               │               │
     │ (最保守值，保证后续事务更大) │               │
     │            │               │               │
     │            │               │ 写入数据       │
     │            │               │ timestamp=157  │
     │            │               │               │
     │            │               │ TT.after(157) │
     │            │               │ 等待...       │
     │            │               │               │
     │            │               │ 开始事务 T2    │
     │            │               │               │
     │            │               │ start_ts2     │
     │            │               │ = TT.now().earliest
     │            │               │ = [158, 165]  │
     │            │               │ 选 158        │
     │            │               │               │
     │            │               │ 158 > 157 ✓   │
     │            │               │ 顺序保证！     │
```

#### 外部一致性保证

```
外部一致性定义:
如果事务 T1 在 T2 开始之前提交，
则 T1 的时间戳必须小于 T2 的时间戳

TrueTime 如何保证:

T1 提交时间: commit_ts1
T2 开始时间: start_ts2

通过 TT.after(commit_ts1):
  start_ts2 >= TT.now().earliest > commit_ts1

数学保证:
  commit_ts1 ∈ [T1_e, T1_l]
  start_ts2 ∈ [T2_e, T2_l]

  TT.after(T1_l) 确保:
  T2_e > T1_l

  因此: start_ts2 > commit_ts1 ✓
```

### 5.5 跨数据中心时钟漂移问题的解决

#### 问题：传统 NTP 无法解决的漂移

```
传统方案的问题:

数据中心 A (北京)     数据中心 B (纽约)
     │                    │
     │ NTP 同步            │ NTP 同步
     │ ↓                  │ ↓
     │ 10:00:00.000       │ 10:00:00.100
     │                    │
     │ 网络延迟变化        │ 网络延迟变化
     │ ↓                  │ ↓
     │ 10:00:00.050       │ 09:59:59.950
     │                    │
     │ 无法预测误差范围！  │

问题：不知道误差有多大，无法保证顺序
```

#### TrueTime 解决方案

```
TrueTime 的方法:

1. 多时间源冗余
   ┌─────────────────────────────────┐
   │ 每台服务器配备：                 │
   │ - GPS 接收机 (多颗卫星)          │
   │ - 原子钟 (本地备份)             │
   │ - NTP (交叉验证)                │
   └─────────────────────────────────┘

2. 持续计算不确定度
   - 实时监测 GPS 信号质量
   - 原子钟漂移率校准
   - 动态调整 ε 值

3. 保守等待策略
   TT.after(t) = 等待直到 earliest > t
   宁可等待，不可出错

结果:
- 全球范围内 ε < 7ms
- 可预测的不确定度
- 严格单调的时间戳
```

### 5.6 局限性

| 问题 | 描述 | 缓解措施 |
|------|------|----------|
| **硬件成本** | GPS+ 原子钟昂贵 | 仅时间服务器需要 |
| **室内/GPS 信号** | 室内无法接收 GPS | 原子钟备份 |
| **等待延迟** | TT.after() 需要等待 | ε 越小等待越短 |
| **单点故障** | 时间服务器故障 | 冗余部署 |

## 6. MongoDB 事务演进

### 6.1 解决的问题

MongoDB 的事务演进反映了对"一致性"认知的变化：

**早期 MongoDB 理念 (4.0 之前)**

```
MongoDB 的设计哲学:
- 大多数 Web 应用不需要强一致性
- 单文档原子性足够满足 90% 场景
- 性能比一致性重要
- 用文档模型避免多文档更新

官方文档曾说:
"MongoDB 不支持事务，用文档模型设计避免事务需求"
```

**为什么改变主意 (4.0+)**

```
现实问题:
- 10% 的场景确实需要多文档事务
- 金融、库存等场景无法用单文档解决
- 用户流失到支持事务的数据库

MongoDB 4.0 (2018) 引入:
- 副本集内多文档事务
MongoDB 4.2 (2019) 引入:
- 分片集群多文档事务
```

---

### 6.2 MongoDB 4.0 之前：单文档原子性

#### 单文档原子性保证

```
MongoDB 单文档操作是原子的:

// 这个操作是原子的
db.users.updateOne(
    { _id: "user123" },
    { $set: { balance: 900 } }
)

// 即使包含多个字段更新，也是原子的
db.users.updateOne(
    { _id: "user123" },
    {
        $set: {
            balance: 900,
            updated_at: new Date()
        },
        $inc: { transaction_count: 1 }
    }
)

保证：
- 其他客户端要么看到更新前，要么看到更新后
- 不会看到中间状态
```

#### 单文档的局限

```
场景：跨文档转账（无法保证原子性）

// ❌ 这不是原子的！
db.accounts.updateOne(
    { user: "A" },
    { $inc: { balance: -100 } }
);
db.accounts.updateOne(
    { user: "B" },
    { $inc: { balance: 100 } }
);

问题:
1. 第一条成功，第二条失败 → A 扣钱 B 未收到
2. 中间状态可见 → 其他客户端看到不一致
3. 无法回滚 → 没有事务概念
```

####  workaround：两阶段更新模式

```
应用层模拟事务（不推荐）:

// 阶段 1：预留
db.accounts.updateOne(
    { user: "A", balance: { $gte: 100 } },
    {
        $set: {
            pending_transfer: { amount: 100, to: "B" }
        }
    }
);

// 检查是否成功
if (result.modifiedCount === 1) {
    // 阶段 2：执行
    db.accounts.updateOne({ user: "A" }, {
        $inc: { balance: -100 },
        $unset: { pending_transfer: "" }
    });
    db.accounts.updateOne({ user: "B" }, {
        $inc: { balance: 100 }
    });
} else {
    // 回滚（其实没执行）
}

问题：
- 复杂，容易出错
- 仍有竞态条件
- 不是真正的事务
```

---

### 6.3 MongoDB 4.0+：多文档事务实现

#### 事务 API

```javascript
// MongoDB 4.0+ 多文档事务示例

const session = db.getMongo().startSession();
session.startTransaction();

try {
    const accounts = db.getCollection('accounts');

    // 扣款
    accounts.updateOne(
        { user: "A" },
        { $inc: { balance: -100 } },
        { session }
    );

    // 收款
    accounts.updateOne(
        { user: "B" },
        { $inc: { balance: 100 } },
        { session }
    );

    // 提交
    session.commitTransaction();
} catch (error) {
    // 回滚
    session.abortTransaction();
} finally {
    session.endSession();
}
```

#### 内部实现原理

```
MongoDB 事务架构:

┌─────────────────────────────────────────┐
│          Client Session                  │
│  (事务会话)                              │
└─────────────────────────────────────────┘
              ↓
┌─────────────────────────────────────────┐
│          Transaction Coordinator         │
│  (事务协调者 - 类似 2PC Coordinator)      │
└─────────────────────────────────────────┘
              ↓
┌─────────────────────────────────────────┐
│          Replica Set                     │
│  ┌─────────┐  ┌─────────┐  ┌─────────┐  │
│  │ Primary │  │Secondary│  │Secondary│  │
│  │ (协调)  │  │         │  │         │  │
│  └─────────┘  └─────────┘  └─────────┘  │
└─────────────────────────────────────────┘

实现细节:
1. 每个事务分配唯一 txnId
2. 操作日志 (oplog) 记录事务标记
3. 提交时一次性应用到所有文档
4. 读写冲突用锁解决
```

#### 事务限制

| 限制 | MongoDB | MySQL |
|------|---------|-------|
| 事务大小 | 最大 16MB | 无硬性限制 |
| 事务时长 | 默认 60 秒超时 | 可配置，通常无限制 |
| 读隔离级别 | 快照读 (Snapshot) | 可配置 (READ COMMITTED/REPEATABLE READ/SERIALIZABLE) |
| 写锁粒度 | 文档级 | 行级 |
| 跨分片事务 | 4.2+ 支持，性能较差 | N/A |

### 6.4 与 MySQL 事务的对比

#### 架构对比

```
MySQL 事务架构:
┌─────────────────────────────────────────┐
│          InnoDB 存储引擎                 │
│  - MVCC 多版本并发控制                   │
│  - Undo Log 回滚                         │
│  - Redo Log 持久化                       │
│  - Row Lock 行锁                         │
└─────────────────────────────────────────┘

MongoDB 事务架构:
┌─────────────────────────────────────────┐
│          WiredTiger 存储引擎             │

│  - 检查点机制                           │

└─────────────────────────────────────────┘
```

#### 性能对比

```
基准测试 (100 万笔转账事务):

| 指标 | MySQL 8.0 | MongoDB 6.0 |
|------|-----------|-------------|
| 事务 TPS | 15,000 | 8,000 |
| 平均延迟 | 2ms | 5ms |
| p99 延迟 | 10ms | 50ms |
| CPU 使用 | 60% | 80% |

结论:
- MySQL 事务性能更优（专门优化）
- MongoDB 事务是"后加"的功能，开销较大
```

#### 使用场景对比

```
MySQL 适合:
✅ 强一致性要求的金融场景
✅ 复杂事务（多表关联）
✅ 长事务（批量处理）
✅ 高并发事务处理

MongoDB 适合:
✅ 文档模型为主的应用
✅ 读多写少的场景
✅ 大部分操作单文档可完成
✅ 需要灵活 Schema

❌ 不适合:
- 高频多文档事务
- 复杂 Join + 事务
- 长事务
```

```
技术演进的螺旋上升:
```

## 分布式事务技术总结

### 技术对比总览

| 技术 | 一致性 | 性能 | 侵入性 | 适用场景 |
|------|--------|------|--------|----------|
| 2PC | 强一致 | 低 | 低 | 传统企业系统 |
| 3PC | 强一致 | 中 | 低 | 较少使用 |
| TCC | 最终一致 | 高 | 高 | 电商、金融 |
| Percolator | 强一致 | 高 | 低 | NewSQL 数据库 |
| Spanner+TrueTime | 强一致 + 外部一致 | 中 | 低 | 全球分布式 |
| MongoDB 事务 | 最终/强一致 | 中 | 低 | 文档型应用 |

---

### 技术选型指南

```
选择决策树:

需要强一致性吗？
├─ 是 → 需要跨数据中心吗？
│       ├─ 是 → Spanner/TiDB (TrueTime/Raft)
│       └─ 否 → MySQL/PostgreSQL (单机/主从)
│
└─ 否 (接受最终一致) → 需要业务控制吗？
        ├─ 是 → TCC (灵活但复杂)
        └─ 否 → 2PC (简单但慢)
```
