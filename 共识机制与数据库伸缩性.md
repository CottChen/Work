# 共识机制与数据库伸缩性

## 3.1 共识机制原理

### 3.1.1 Paxos 算法

**解决的问题**：为什么需要 Paxos

在分布式系统中，多个节点需要对某个值达成一致。网络可能延迟、节点可能故障、消息可能丢失或重复。Paxos 解决了"如何在不可靠的环境中让节点达成共识"的问题。

典型场景：
- 数据库主节点选举
- 配置信息同步
- 事务提交决策

**原理**：核心工作机制

Paxos 包含三种角色：
- **Proposer（提议者）**：提出议案（要存储的值）
- **Acceptor（接受者）**：投票决定是否接受议案
- **Learner（学习者）**：学习最终达成的决议

Basic Paxos 分为两个阶段：

```
阶段 1：Prepare（准备阶段）
1. Proposer 选择一个提案编号 n，向多数 Acceptors 发送 Prepare(n) 请求
2. Acceptor 收到 Prepare(n) 后：
   - 如果 n 大于它已响应的所有 Prepare 请求编号
   - 回复 Promise，承诺不接受编号小于 n 的提案
   - 如果之前已接受过提案，回复已接受的最大编号提案 (n', value)

阶段 2：Accept（接受阶段）
1. Proposer 收到多数 Acceptors 的 Promise 后：
   - 如果任何 Acceptor 回复了已接受的提案，使用其中编号最大的 value
   - 否则，可以使用自己提出的 value
   - 向这些 Acceptors 发送 Accept(n, value) 请求
2. Acceptor 收到 Accept(n, value) 后：
   - 如果不违背之前的 Promise，接受该提案并回复 Accepted
3. 当多数 Acceptors 接受后，提案被批准
```

**详细例子**：

考虑一个 5 节点集群 (A1-A5)，需要就"是否提交事务 T"达成一致。

```
时序图：Basic Paxos 流程

Proposer P1              Acceptors(A1-A5)              Learner
    |                           |                         |
    |---- Prepare(1) --------->|                         |
    |                           |                         |
    |<--- Promise(1,null) -----| A1, A2, A3 (多数)        |
    |                           |                         |
    |---- Accept(1,"Commit") ->|                         |
    |                           |                         |
    |<-- Accepted(1,"Commit") -| A1, A2, A3               |
    |                           |                         |
    |                           |------- 通知 ------------>|
    |                           |                         |
    |                     值确定为"Commit"                 |
```

**数据分布示例**：

```
初始状态：所有节点值为 null

节点 A1: null
节点 A2: null
节点 A3: null
节点 A4: null
节点 A5: null

步骤 1：P1 发送 Prepare(1)
A1, A2, A3 回复 Promise(1, null)

步骤 2：P1 发送 Accept(1, "X=100")
A1, A2, A3 回复 Accepted

步骤 3：P1 故障，P2 不知道结果，发送 Prepare(2)
A4, A5 回复 Promise(2, null)
A3 回复 Promise(2, 已接受"X=100")  <- 关键：告知已接受的值

步骤 4：P2 必须使用"X=100"发送 Accept(2, "X=100")
这保证了即使 Proposer 变更，已批准的值不会改变
```

**与其他机制的对比**：

| 特性 | Basic Paxos | Multi Paxos | Raft |
|------|-------------|-------------|------|
| 角色 | Proposer/Acceptor/Learner | 同左 | Leader/Follower/Candidate |
| 阶段 | 两阶段 | 一次 Prepare 后单阶段 | 领导者单阶段 |
| 性能 | 较低 | 高 | 高 |
| 理解难度 | 困难 | 中等 | 容易 |
| 应用 | 理论研究 | Spanner | etcd, CockroachDB |

**局限性**：
- 性能问题：每个提案都需要两阶段提交
- 活锁问题：多个 Proposer 可能互相冲突，导致无法达成决议
- 复杂性：完整实现非常复杂，需要处理边界情况

### 3.1.2 Multi Paxos 优化

**解决的问题**：为什么需要 Multi Paxos

Basic Paxos 每个提案都需要两阶段，性能低下。实际场景中，通常需要就一系列值达成共识（如日志复制）。Multi Paxos 通过选举一个稳定的 Leader 来优化这一过程。

**原理**：

```
优化策略：
1. 选举一个 Leader 作为唯一的 Proposer
2. Leader 只需执行一次 Prepare 阶段建立领导地位
3. 后续提案直接进入 Accept 阶段
4. 如果 Leader 故障，重新选举
```

**流程**：

```
Leader 选举阶段（Prepare）:
Leader -> Acceptors: Prepare(LeaderID)
Acceptors -> Leader: Promise(LeaderID)

正常提案阶段（Accept）:
Leader -> Acceptors: Accept(LeaderID, value_i)
Acceptors -> Leader: Accepted
```

**在数据库中的应用**：

| 数据库 | Paxos 变体 | 用途 |
|--------|-----------|------|
| Google Spanner | Multi Paxos | 真全局事务、副本同步 |
| TiKV | Raft (Paxos 变种) | 分布式事务日志 |
| etcd | Raft | 配置存储 |

### 3.1.3 Raft 算法

**解决的问题**：为什么需要 Raft

Paxos 虽然正确但难以理解和实现。Raft 的设计目标是"易于理解"，通过清晰的职责分离和直观的规则，让分布式共识变得可教学、可实现。

**原理**：核心工作机制

Raft 将共识问题分解为三个子问题：
1. **领导者选举**：集群需要一个领导者来处理所有请求
2. **日志复制**：领导者将日志复制到追随者
3. **安全性**：保证已提交的日志不会丢失

**节点状态**：
- **Follower（追随者）**：被动响应，只能响应 Leader 和 Candidate 的请求
- **Candidate（候选人）**：发起选举，请求投票
- **Leader（领导者）**：处理所有客户端请求，复制日志到 Follower

**详细例子 - Raft 选举时序图**：

```

初始状态：所有节点为 Follower，无 Leader

时间线 ──────────────────────────────────────────────>

t0: Leader 故障，集群进入选举超时
┌─────────┐  ┌─────────┐  ┌─────────┐
│ Node A  │  │ Node B  │  │ Node C  │
│Follower │  │Follower │  │Follower │
│timeout! │  │timeout  │  │timeout  │

**日志复制流程**：

```
步骤分解：

1. Leader 接收客户端命令
   Leader: 收到 SET x=100

2. Leader 在本地日志添加新条目
   Node A 日志：[term1:CMD1, term1:CMD2, term5:SET x=100]

3. Leader 并行发送 AppendEntries 到所有 Follower
   包含：新条目的索引、term、内容、前一条目信息

4. Follower 一致性检查
   - 检查 prevLogIndex 和 prevLogTerm 是否匹配
   - 如果匹配，追加新条目
   - 如果不匹配，拒绝请求

5. Follower 回复 ACK

6. Leader 统计 ACK
   - 收到多数 ACK 后，标记条目为"已提交"
   - 应用到本地状态机
   - 下次心跳通知 Follower 提交

7. Follower 应用已提交的条目
```

**安全性保证**：

| 保证 | 实现机制 |
|------|----------|
| 选举限制 | Candidate 必须包含所有已提交条目才能当选 |
| 日志匹配 | 通过 prevLogIndex/Term 保证日志一致性 |
|  Leader 追加 | 只有 Leader 能写入，Follower 只能复制 |
| 状态机安全 | 已提交的条目在所有后续 Leader 日志中 |

**在数据库中的应用**：

| 数据库 | 使用场景 |
|--------|----------|
| etcd | 核心共识协议，存储所有键值对 |
| CockroachDB | 每个 Range 使用独立的 Raft 组 |
| TiKV | Region 级别的 Raft 复制 |
| Consul | 服务发现和配置存储 |

### 3.1.4 ZAB (ZooKeeper Atomic Broadcast)

**解决的问题**：为什么需要 ZAB

ZooKeeper 需要高吞吐量的有序消息广播，同时保证崩溃恢复后的数据一致性。Paxos 和 Raft 虽然能达成共识，但 ZAB 针对 ZooKeeper 的使用场景做了专门优化。

**原理**：

ZAB 包含两个模式：
1. **消息广播模式**：正常状态下 Leader 广播事务
2. **崩溃恢复模式**：选举新的 Leader 并同步数据

**与 Paxos/Raft 的区别**：

| 特性 | ZAB | Raft | Paxos |
|------|-----|------|-------|
| 设计目标 | 高吞吐广播 | 易理解 | 理论正确性 |
| 领导者角色 | 协调者 | 唯一写入者 | 可以是多 Proposer |
| 崩溃恢复 | 显式恢复阶段 | 隐式通过选举 | 隐式 |
| 消息顺序 | 全局 FIFO | 日志顺序 | 提案顺序 |
| 应用场景 | ZooKeeper | etcd, CockroachDB | Spanner |

**ZAB 流程**：

```
崩溃恢复阶段：
1. 集群启动或 Leader 故障后进入恢复模式
2. 所有节点参与选举，选出具有最新数据的节点作为 Leader
3. Leader 等待多数 Follower 连接
4. Leader 与 Follower 同步数据（差异同步）
5. 同步完成后进入消息广播模式

消息广播阶段：
1. Leader 接收事务请求
2. 生成带全局有序 ID 的事务提案 (zxid)
3. 广播到所有 Follower
4. Follower 回复 ACK
5. 收到多数 ACK 后提交
6. 通知 Follower 提交
```

**在 ZooKeeper 中的应用**：

```
ZooKeeper 数据模型：
/zookeeper/quota   - 配额配置
/app/config        - 应用配置
/app/leader        - 领导者节点（临时节点）
/app/locks/lock1   - 分布式锁

ZAB 保证：
- 所有节点看到相同的更新顺序
- 一旦写入成功，不会丢失（除非多数节点故障）
- 客户端读取可能看到旧值（最终一致性）
```

**Kafka 与 ZooKeeper**：

```
Kafka 使用 ZooKeeper 存储：
- Broker 元数据（哪些 Broker 存活）
- Topic 配置（分区数、副本因子）
- Consumer Group 偏移量（旧版本）
- Controller 选举（哪个 Broker 是 Controller）

Kafka 不直接使用 ZAB 复制数据：
- Kafka 数据复制使用自己的 ISR (In-Sync Replicas) 协议
- ZooKeeper 仅用于元数据和协调
```

**局限性**：
- 写入性能受限于 Leader
- 恢复阶段可能较长（大数据量时）
- 需要奇数节点（通常 3 或 5 个）

## 3.2 一致性哈希 (Consistent Hashing)

**解决的问题**：为什么需要一致性哈希

在传统哈希方案中，数据映射到节点使用 `hash(key) % N`。当节点数 N 变化时（扩容/缩容），几乎所有数据都需要重新映射，导致大规模数据迁移。

一致性哈希解决的核心问题：
- 节点增减时，最小化数据迁移
- 无需全局协调，节点可独立加入/离开
- 数据均匀分布，避免热点

**原理**：核心工作机制

一致性哈希将节点和数据都映射到一个环形空间：

```
一致性哈希环：
- 环的范围：0 到 2^32-1（或 0 到 360 度）
- 节点：hash(node_id) 映射到环上某点
- 数据：hash(key) 映射到环上某点
- 归属规则：数据属于顺时针方向第一个节点
```

**详细例子 - 一致性哈希环示例**：

```
步骤 1：构建哈希环（4 个节点）

假设使用简单的 hash 函数，环范围 0-360

节点哈希值：
- Node A: hash("A") = 45°
- Node B: hash("B") = 150°
- Node C: hash("C") = 270°
- Node D: hash("D") = 330°

哈希环示意：

                    0°/360°
                     │
        Node D (330°)│   Node A (45°)
              ╲      │      ╱
               ╲     │     ╱
                ╲    │    ╱
    ──────────────●───●──────────────
                 ╱    │    ╲
                ╱     │     ╲
               ╱      │      ╲
              ╱       │       ╲
             ╱        │        ╲
        270° ● Node C │ Node B 150°
```

```
步骤 2：数据映射

数据项及其哈希值：
- "user:1"   → hash = 30°   → 顺时针第一个节点是 A(45°) → 存储到 Node A
- "user:2"   → hash = 100°  → 顺时针第一个节点是 A(45°) → 存储到 Node A
- "user:3"   → hash = 200°  → 顺时针第一个节点是 C(270°) → 存储到 Node C
- "user:4"   → hash = 300°  → 顺时针第一个节点是 D(330°) → 存储到 Node D
- "user:5"   → hash = 350°  → 顺时针第一个节点是 A(45°) → 存储到 Node A（绕回）

数据分布：
Node A: user:1, user:2, user:5 (3 个)
Node B: 无 (0 个)  ← 分布不均！
Node C: user:3 (1 个)
Node D: user:4 (1 个)
```

```
步骤 3：新增节点（扩容）

新增 Node E，hash("E") = 200°

                    0°/360°
                     │
        Node D (330°)│   Node A (45°)
              ╲      │      ╱
               ╲     │     ╱
                ╲    │    ╱
    ──────────────●───●──────────────
                 ╱    │    ╲
                ╱     │     ╲
               ╱      │      ╲
         Node E(200°) │      ╲
              ●───────●───────●
             ╱        │        ╲
        270° ● Node C │ Node B 150°

影响范围：
- Node E 只影响其逆时针方向到前一个节点之间的数据
- 即 (150°, 200°] 范围内的数据需要迁移到 Node E
- 具体：user:3 从 Node C 迁移到 Node E

其他节点数据不受影响！
```

**虚拟节点概念**：

```
问题：上面例子中 Node B 没有分配到任何数据

解决：使用虚拟节点（Virtual Node）

每个物理节点映射多个虚拟节点：
- Node A → A_1(30°), A_2(120°), A_3(250°)
- Node B → B_1(60°), B_2(180°), B_3(300°)
- Node C → C_1(90°), C_2(210°), C_3(340°)

带虚拟节点的环：

               0°/360°
                 │
       A_1(30°)  │  B_1(60°)
           ╲     │     ╱
            ╲    │    ╱
        C_1 ●───●───● B_2(180°)
    (90°) ╱   A_2 │    ╲
         ╱ (120°) │     ╲
        ╱         │      ╲
       ●──────────●───────●
    C_2(210°)   A_3(250°) B_3(300°)
                         ╱
                        ╱
                    C_3(340°)

优势：
1. 数据分布更均匀
2. 节点增减影响更分散
3. 可根据节点性能调整虚拟节点数量（性能强的节点更多虚拟节点）
```

**数据分布与再平衡**：

```
再平衡触发条件：
1. 新增节点：新节点承担部分数据
2. 节点下线：该节点数据迁移到后继节点
3. 节点负载不均：手动触发虚拟节点迁移

再平衡算法：
1. 计算每个节点的负载（数据量/请求量）
2. 如果负载差异超过阈值（如 20%）
3. 将高负载节点的部分虚拟节点迁移到低负载节点
4. 仅迁移受影响的数据

示例：
初始：Node A 负载 60%，Node B 负载 20%，Node C 负载 20%

再平衡：
- 从 Node A 迁移 2 个虚拟节点到 Node B
- 迁移对应的数据
- 结果：Node A 40%，Node B 35%，Node C 25%
```

**与其他机制的对比**：

| 特性 | 传统哈希 | 一致性哈希 | 一致性哈希 + 虚拟节点 |
|------|----------|------------|---------------------|
| 扩容影响 | O(N) 全部迁移 | O(N/M) 部分迁移 | O(N/M) 更细粒度 |
| 数据均匀性 | 均匀 | 可能不均 | 均匀 |
| 实现复杂度 | 简单 | 中等 | 中等 |
| 应用场景 | 单机哈希表 | 分布式缓存 | Dynamo, Cassandra |

**在分布式数据库中的应用**：

| 数据库 | 一致性哈希实现 |
|--------|---------------|
| DynamoDB | 一致性哈希环，虚拟节点 |
| Cassandra | 令牌环（Token Ring），虚拟节点 |
| Redis Cluster | 16384 个槽（简化的一致性哈希） |
| Couchbase | vBucket 机制 |

**局限性**：
- 数据分布可能不均匀（需要虚拟节点）
- 范围查询效率低（数据随机分布）
- 节点故障时影响范围不可预测

## 3.3 Redis Cluster 扩容机制

**解决的问题**：为什么需要 Redis Cluster 分片

单个 Redis 实例的内存容量和吞吐量有限。当数据量增长或 QPS 过高时，需要水平扩展到多个节点。Redis Cluster 提供了：
- 自动分片（16384 个槽）
- 节点故障自动检测
- 在线扩容/缩容能力

**原理**：16384 个槽的概念

Redis Cluster 将整个键空间划分为 16384 个槽（slot）：
- 槽编号：0 - 16383
- 槽分配：`slot = CRC16(key) % 16384`
- 节点管理：每个节点负责一部分槽
- 集群配置：每个节点记录所有槽的分布

**为什么是 16384**：
- 2^14 = 16384，适中的大小
- 足够小：槽信息可以用紧凑的位图表示（2KB）
- 足够大：可以精细分配，支持大规模集群

**详细例子 - 节点如何分配槽**：

```
3 节点集群初始状态：

Node A (127.0.0.1:7000): 槽 0 - 5460    (共 5461 个槽)
Node B (127.0.0.1:7001): 槽 5461 - 10922 (共 5462 个槽)
Node C (127.0.0.1:7002): 槽 10923 - 16383 (共 5461 个槽)

槽分布示意：

0                    5461               10923              16383
│                      │                   │                  │
├──────────────────────┼───────────────────┼──────────────────┤
│      Node A          │      Node B       │      Node C      │
│    (0-5460)          │  (5461-10922)     │ (10923-16383)    │
└──────────────────────┴───────────────────┴──────────────────┘

数据分布示例：
key="user:1"  → CRC16("user:1") % 16384 = 3028  → Node A
key="user:2"  → CRC16("user:2") % 16384 = 8500  → Node B
key="user:3"  → CRC16("user:3") % 16384 = 15000 → Node C
```

**详细例子 - Redis Cluster 扩容流程（3 节点→5 节点）**：

```

```

```

**与其他机制的对比**：

| 特性 | Redis Cluster | Cassandra | MongoDB |

**客户端如何处理重定向**：

```
场景：客户端连接 Node A，但要访问的 key 在 Node B

客户端请求流程：

1. 客户端发送：GET user:2 → Node A
2. Node A 计算：CRC16("user:2") % 16384 = 8500
3. Node A 发现：槽 8500 属于 Node B
4. Node A 返回：MOVED 8500 127.0.0.1:7001
5. 客户端解析 MOVED 错误
6. 客户端更新本地槽映射缓存
7. 客户端重新发送：GET user:2 → Node B
8. Node B 返回数据

伪代码：

def redis_cluster_get(key):
    slot = CRC16(key) % 16384
    node = slot_map[slot]  # 本地缓存的槽映射

    try:
        return node.get(key)
    except MOVED Error as e:
        # 更新槽映射
        slot_map[e.slot] = e.new_node
        # 重试
        return e.new_node.get(key)
    except ASK Error as e:
        # 迁移中的临时重定向
        return e.new_node.asking_get(key)
```

```
MOVED vs ASK 重定向：

MOVED 重定向（永久）：
- 槽已完成迁移
- 客户端应更新本地缓存
- 后续请求直接发送到新节点

ASK 重定向（临时）：
- 槽正在迁移中
- 客户端发送 ASKING 命令到目标节点
- 目标节点允许访问尚未完全迁移的 key
- 客户端不需要更新本地缓存

ASK 流程：
客户端 → 源节点：GET key
源节点 → 客户端：ASK 1234 127.0.0.1:7003
客户端 → 目标节点：ASKING
客户端 → 目标节点：GET key
目标节点 → 客户端：数据
```

**局限性**：
- 不支持多键事务（跨槽操作）
- 槽再平衡期间可能影响性能
- 客户端需要支持集群协议
- 运维复杂度增加

## 3.4 MongoDB 分片与副本集

**解决的问题**：为什么需要 MongoDB 分片与副本集

- **副本集**：解决单点故障，提供高可用性
- **分片**：解决单节点存储和性能瓶颈，支持水平扩展

MongoDB 将两者结合：每个分片本身是一个副本集，既有高可用又可水平扩展。

---

### 3.4.1 副本集 (Replica Set)

**原理**：

MongoDB 副本集由 3-50 个节点组成：
- **Primary（主节点）**：处理所有写操作，复制数据到 Secondary
- **Secondary（从节点）**：复制 Primary 的数据，可提供读服务
- **Arbiter（仲裁者）**：不参与数据复制，只参与选举投票

**详细例子 - 主从选举**：

```
副本集配置：
- Primary: mongo-1:27017
- Secondary: mongo-2:27017, mongo-3:27017
- Arbiter: mongo-4:27017

选举触发条件：
1. Primary 宕机
2. Primary 网络分区
3. 手动步进（rs.stepDown()）
4. 初始化新副本集

选举过程（基于 Raft 的变种）：

时间线 ──────────────────────────────────────────────>

t0: 正常运行
┌─────────┐  ┌─────────┐  ┌─────────┐
│mongo-1  │  │mongo-2  │  │mongo-3  │
│Primary  │  │Secondary│  │Secondary│
│心跳正常 │  │心跳正常 │  │心跳正常 │
└─────────┘  └─────────┘  └─────────┘

t1: Primary 故障（宕机或网络分区）
┌─────────┐  ┌─────────┐  ┌─────────┐
│  DOWN   │  │mongo-2  │  │mongo-3  │
│         │  │心跳超时 │  │心跳超时 │
│         │  │开始选举 │  │等待选举 │
└─────────┘  └────┬────┘  └────┬────┘
                  │             │
                  │ 超时到达    │ 超时到达
                  │ (10 秒)     │ (10 秒)
                  ▼             ▼

t2: mongo-2 成为 Candidate
┌─────────┐  ┌─────────┐  ┌─────────┐
│         │  │mongo-2  │  │mongo-3  │
│         │  │Candidate│  │Secondary│
│         │  │Term=10  │  │Term=9   │
│         │  │投票自己 │  │         │
└─────────┘  └────┬────┘  └────┬────┘
                  │             │
                  │ requestVote │
                  ├────────────>│
                  │             │
                  │             │ 比较 Term
                  │             │ mongo-2 Term 更新
                  │             │ 检查日志
                  │             │ 投票给 mongo-2
                  │<────────────┤
                  │ voteGranted │
                  │             │
                  ▼             │
获得多数票 (2/3)               │
成为 Primary                   │
                  │             │
                  │ 通知        │
                  ├────────────>│
                  │             │ 更新为 Term=10

t3: 选举完成
┌─────────┐  ┌─────────┐  ┌─────────┐
│         │  │mongo-2  │  │mongo-3  │
│  DOWN   │  │Primary  │  │Secondary│
│         │  │Term=10  │  │Term=10  │
└─────────┘  └─────────┘  └─────────┘
```

**oplog 复制**：

```
oplog (操作日志) 是副本集复制的核心

oplog 结构：
{
  _id: ObjectId("..."),
  ts: Timestamp(1699999999, 1),  // 时间戳 + 操作序号
  h: 123456789,                   // 操作唯一标识
  op: "i",                        // 操作类型：i=insert, u=update, d=delete, c=command
  ns: "test.users",               // 命名空间（数据库。集合）
  o: { _id: 1, name: "Alice" }    // 操作详情
}

复制流程：

Primary (mongo-1):
1. 客户端发送：db.users.insert({name: "Alice"})
2. Primary 验证操作合法
3. 写入 oplog（特殊 capped collection：local.oplog.rs）
4. 应用到本地数据集
5. 返回成功给客户端

Secondary (mongo-2, mongo-3):
1. 尾随 Primary 的 oplog（tailing oplog）
2. 获取新的 oplog 条目
3. 在本地重放操作
4. 更新本地数据

同步机制：
- Secondary 持续拉取 Primary 的 oplog
- 使用 getMore 游标保持长连接
- 如果落后太多，触发初始同步（全量复制）
```

```
oplog 复制时序图：

Client    Primary(mongo-1)    Secondary(mongo-2)
   │              │                    │
   │──insert───>  │                    │
   │              │                    │
   │              │─ 写入 oplog ───────│
   │              │─ 应用数据 ─────────│
   │              │                    │ 拉取 oplog
   │              │───────────────────>│
   │              │                    │─ 重放操作
   │              │                    │─ 应用数据
   │              │                    │
   │<──成功 ─────  │                    │
```

**读偏好设置 (Read Preference)**：

```
MongoDB 允许客户端配置读操作路由到哪个节点：

1. primary（默认）
   - 只从 Primary 读取
   - 保证读取最新数据
   - Primary 故障时读取失败

2. primaryPreferred
   - 优先从 Primary 读取
   - Primary 不可用时从 Secondary 读取
   - 可能读到旧数据

3. secondary
   - 只从 Secondary 读取
   - 适用于报表、分析等读旧数据可接受的场景
   - Primary 故障不影响读取

4. secondaryPreferred
   - 优先从 Secondary 读取
   - Secondary 不可用时从 Primary 读取

5. nearest
   - 读取网络延迟最低的节点
   - 基于 ping 时间选择

使用示例：
db.getMongo().setReadPref("secondary")
db.users.find({status: "active"})

带标签的读偏好：
db.getMongo().setReadPref(
  "secondary",
  [{dc: "sh", rack: "r1"}]  // 只从指定数据中心读取
)
```

### 3.4.2 分片 (Sharding)

**原理**：

MongoDB 分片集群包含三种组件：
- **Shard（分片）**：存储实际数据，每个 Shard 是一个副本集
- **Mongos（路由）**：查询路由，客户端连接入口
- **Config Server（配置服务器）**：存储集群元数据

**架构示意**：

```
                    ┌─────────────┐
                    │   Mongos    │
                    │  (路由器)   │
                    └──────┬──────┘
                           │
         ┌─────────────────┼─────────────────┐
         │                 │                 │
         ▼                 ▼                 ▼
┌─────────────────┐ ┌─────────────────┐ ┌─────────────────┐
│   Shard 1       │ │   Shard 2       │ │   Shard 3       │
│ ┌─────────────┐ │ │ ┌─────────────┐ │ │ ┌─────────────┐ │
│ │ Primary     │ │ │ │ Primary     │ │ │ │ Primary     │ │
│ │ Sec         │ │ │ │ Sec         │ │ │ │ Sec         │ │
│ └─────────────┘ │ │ └─────────────┘ │ │ └─────────────┘ │
└─────────────────┘ └─────────────────┘ └─────────────────┘
         ▲                 ▲                 ▲
         │                 │                 │
         └─────────────────┼─────────────────┘
                           │
                  ┌────────┴────────┐
                  │  Config Server  │
                  │    (副本集)     │
                  └─────────────────┘
```

**片键选择**：

片键 (Shard Key) 是分片的核心，决定数据如何分布。

| 片键类型 | 说明 | 适用场景 | 注意事项 |
|----------|------|----------|----------|
| 单字段片键 | 使用一个字段的值 | 简单场景 | 避免单调递增字段 |
| 复合片键 | 多个字段组合 | 复杂查询 | 字段顺序很重要 |
| 哈希片键 | 对字段值哈希 | 均匀分布 | 不支持范围查询 |
| 分片键范围 | 按值范围分片 | 范围查询 | 可能产生热点 |

**片键选择示例**：

```
场景：用户订单集合

坏的选择：
 shardKey: { createdAt: 1 }
 问题：新订单总是写入最后一个分片，产生写入热点

好的选择：
 shardKey: { userId: "hashed" }
 优点：用户订单均匀分布到各分片
 缺点：范围查询（如查询某时间段订单）效率低

折中选择：
 shardKey: { userId: 1, createdAt: 1 }
 优点：同一用户订单在一起，可范围查询时间
 缺点：需要选择合适的 userId 基数
```

**数据分布策略**：

```
MongoDB 数据分布单位：Chunk（块）

Chunk 定义：
- 每个 Chunk 包含片键范围的一个子集
- 默认大小：64MB（可配置）
- Chunk 是分片迁移的最小单位

范围分片示例：
片键：userId (1-10000)

Shard A: userId [1, 2500)
Shard B: userId [2500, 5000)
Shard C: userId [5000, 7500)
Shard D: userId [7500, 10000)

每个 Chunk 约 64MB，可能包含数百到数千个文档

哈希分片示例：
片键：userId (hashed)

Shard A: hash [0, 2^31)
Shard B: hash [2^31, 2^32)
...

哈希分片自动均匀分布，但无法高效执行范围查询
```

**均衡器工作原理**：

```
均衡器 (Balancer) 是 MongoDB 的后台进程，负责自动再平衡

触发条件：
1. 分片间 Chunk 数量差异超过阈值
2. 新分片加入集群
3. 片键范围发生变化

均衡器运行流程：

1. 检测不平衡
   Config Server 检查各分片的 Chunk 分布

2. 选择迁移 Chunk
   选择最大分片上的一个 Chunk 进行迁移

3. 执行迁移
   - 源分片标记 Chunk 为"迁移中"
   - 目标分片开始复制数据
   - 复制完成后，更新元数据
   - 源分片删除已迁移的数据

4. 继续检查
   重复直到各分片 Chunk 数差异在阈值内

均衡器状态查看：
db.settings.find({ _id: "balancer" })
db.getSiblingDB("config").chunks.aggregate([
  { $group: { _id: "$shard", count: { $sum: 1 } } }
])
```

**MongoDB 分片对比表格**：

| 特性 | 范围分片 | 哈希分片 |
|------|----------|----------|
| 适用场景 | 范围查询（如时间范围、ID 范围） | 均匀分布写入负载 |
| 热点问题 | 可能出现（单调递增片键） | 有效避免 |
| 扩容复杂度 | 较复杂（需要迁移连续范围） | 简单（自动均匀分布） |
| 查询效率 | 范围查询高效 | 范围查询需散射到所有分片 |
| 数据局部性 | 相关数据在同一分片 | 数据随机分布 |
| 片键要求 | 需要选择基数合适的字段 | 任何字段都可 |

**扩容流程**：

```
场景：从 3 分片扩展到 5 分片

初始状态：
Shard A: 1000 Chunks
Shard B: 1000 Chunks
Shard C: 1000 Chunks
总计：3000 Chunks

步骤 1：添加新分片
rs.addShard("shard-d/rs-d/mongo-d1:27017")
rs.addShard("shard-e/rs-e/mongo-e1:27017")

步骤 2：均衡器自动开始迁移
- 检测到 Shard A、B、C 各有 1000 Chunks
- Shard D、E 有 0 Chunks
- 开始从 A、B、C 向 D、E 迁移

步骤 3：迁移过程
每个 Chunk 迁移：
1. 源分片锁定 Chunk（禁止写入）
2. 目标分片复制数据
3. 更新 Config Server 元数据
4. 源分片解锁并删除数据

步骤 4：达到平衡
Shard A: 600 Chunks
Shard B: 600 Chunks
Shard C: 600 Chunks
Shard D: 600 Chunks
Shard E: 600 Chunks
总计：3000 Chunks

扩容命令：
use admin
db.runCommand({ addShard: "shard-d/rs-d/mongo-d1:27017" })
db.runCommand({ addShard: "shard-e/rs-e/mongo-e1:27017" })

查看分片状态：
db.printShardingStatus()
```

**局限性**：
- 片键选择后难以更改
- 跨分片查询性能较差
- 均衡器运行占用资源
- 小数据量不建议分片

## 3.5 分片再平衡

**解决的问题**：为什么需要分片再平衡

分片再平衡解决的核心问题：
- 数据分布不均匀（某些分片负载过高）
- 新增/删除分片后重新分配数据
- 业务增长导致的数据倾斜
- 节点故障后的数据恢复

**什么时候需要再平衡**：

```
触发条件：

1. 新分片加入集群
   - 新分片初始数据量为 0
   - 需要从现有分片迁移数据

2. 分片负载不均
   - 最大分片与最小分片差异超过阈值
   - 通常阈值：Chunk 数差异 > 8

3. 片键范围变化
   - 数据增长导致某些范围数据量激增
   - 需要拆分过大的 Chunk

4. 分片下线
   - 需要将该分片的数据迁移到其他分片
   - 通常是手动操作（removeShard）

5. 手动触发
   - 管理员手动启动平衡
   - 更改片键后重新分片
```

**再平衡算法**：

```
MongoDB 均衡器算法：

1. 采样阶段
   均衡器定期（默认 6 秒）检查各分片的 Chunk 分布

2. 决策阶段
   计算每个分片的 Chunk 数量
   找出最大分片 (maxShard) 和最小分片 (minShard)

   如果不平衡条件满足：
   (maxShard.chunks - minShard.chunks) > threshold
   则触发迁移

3. 选择迁移 Chunk
   从 maxShard 选择一个 Chunk：
   - 优先选择边界 Chunk（更容易迁移）
   - 避免选择频繁访问的热点 Chunk
   - 考虑迁移成本（数据量大小）

4. 执行迁移
   迁移过程：

   ┌─────────────┐                    ┌─────────────┐
   │  Source     │                    │  Target     │
   │   Shard     │                    │   Shard     │
   └──────┬──────┘                    └──────┬──────┘
          │                                  │
          │  1. 标记 Chunk 为迁移中           │
          │  (克隆阶段开始)                   │
          │─────────────────────────────────>│
          │                                  │
          │  2. 全量复制数据                  │
          │─────────────────────────────────>│
          │                                  │
          │  3. 同步增量 oplog               │
          │─────────────────────────────────>│
          │                                  │
          │  4. 更新 Config Server 元数据     │
          │<────────────────────────────────>│
          │                                  │
          │  5. 切换路由（Mongos 更新）        │
          │                                  │
          │  6. 删除源分片数据               │
          │<─────────────────────────────────│
          │                                  │
   └─────────────┘                    └─────────────┘

5. 迭代
   重复上述过程直到达到平衡状态
```

**如何保证不影响在线服务**：

```
1. 增量同步机制

迁移过程中，源分片继续处理读写请求：
- 写操作：正常写入，同时记录 oplog
- 读操作：从源分片读取（迁移完成后切换）

目标分片同步流程：
阶段 1：全量复制
  - 后台复制数据，不阻塞源分片
  - 源分片继续服务

阶段 2：增量追赶
  - 尾随源分片的 oplog
  - 实时同步新写入

阶段 3：短暂切换
  - 锁定 Chunk（毫秒级）
  - 更新元数据
  - 切换路由
  - 解锁

2. 限流控制

均衡器自动限制迁移速度：
- 每次只迁移一个 Chunk
- 两次迁移之间有间隔
- 检测集群负载，高负载时暂停

配置参数：
db.settings.updateOne(
  { _id: "balancer" },
  { $set: { active: true, mode: "full" } }
)

模式说明：
- full: 正常平衡
- onlyChunk: 只迁移 Chunk，不分拆合并
- never: 禁用平衡

3. 读写分离

迁移期间：
- 读请求：继续从源分片读取
- 写请求：继续写入源分片
- 目标分片只接收复制数据

元数据切换后：
- Mongos 路由更新
- 新请求发送到目标分片
- 源分片清理数据

4. 原子性保证

Chunk 迁移是原子操作：
- 要么完全成功
- 要么完全失败（回滚）
- 不会出现数据不一致

失败处理：
- 迁移失败时，Chunk 回到源分片
- 元数据回滚
- 记录失败原因

5. 客户端无感知

对客户端透明：
- 客户端只连接 Mongos
- 不感知底层分片变化
- 查询自动路由到正确分片

例外情况：
- 迁移切换瞬间可能有短暂延迟
- 片键查询可能受影响
- 跨分片事务可能失败（需重试）
```

**与其他机制的对比**：

| 特性 | MongoDB | Cassandra | Redis Cluster |
|------|---------|-----------|---------------|
| 再平衡单位 | Chunk (64MB) | Token Range | Slot |
| 触发方式 | 自动/手动 | 自动 | 手动 |
| 迁移影响 | 低 | 低 | 中等 |
| 数据一致性 | 强一致 | 最终一致 | 最终一致 |

**再平衡监控**：

```
查看平衡状态：
db.printShardingStatus()

查看迁移历史：
use config
db.changelog.find().sort({ time: -1 }).limit(10)

手动触发平衡：

禁用平衡：

```

**局限性**：
- 均衡器运行期间占用集群资源
- 大数据量迁移可能影响性能
- 片键选择不当会导致频繁再平衡
- 跨分片查询性能下降

## 总结

本章介绍了分布式数据库的核心技术：

1. **共识机制**（Paxos、Raft、ZAB）：解决分布式系统中数据一致性问题
2. **一致性哈希**：解决数据均匀分布和弹性伸缩问题
3. **Redis Cluster**：基于槽的分片和扩容机制
4. **MongoDB 分片**：副本集 + 分片的完整解决方案
5. **分片再平衡**：保证数据均匀分布和在线服务

这些技术共同构成了现代分布式数据库的基础架构。
